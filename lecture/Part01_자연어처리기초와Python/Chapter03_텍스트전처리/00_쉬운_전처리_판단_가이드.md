# Chapter 3 쉬운 전처리 판단 가이드

## 전처리는 청소가 아니라 기준 통일이다

텍스트 전처리를 “지저분한 글자를 지우는 작업”으로만 생각하면 중요한 정보까지 없앨 수 있습니다. 전처리의 진짜 목적은 같은 의미를 가능한 한 같은 형태로 표현하고, 모델이 불필요한 차이에 흔들리지 않게 만드는 것입니다.

예를 들어 감성 분석에서 `좋아요`, `좋아용`, `좋아요!!!`를 모두 비슷한 표현으로 볼 수 있습니다. 그러나 스팸 탐지에서는 반복 느낌표가 중요한 신호일 수 있습니다. 따라서 전처리는 작업 목적에 따라 달라집니다.

## 먼저 질문할 다섯 가지

1. 최종 과제는 분류, 검색, 요약 중 무엇인가?
2. 제거하려는 문자가 의미를 갖는가?
3. 대소문자·띄어쓰기 차이를 합쳐도 되는가?
4. 숫자와 고유명사를 보존해야 하는가?
5. 처리 전 원문을 다시 확인할 수 있는가?

원문 열과 정제 열을 함께 보관하면 전처리 오류를 추적하기 쉽습니다.

## 정규표현식을 필터로 이해하기

정규표현식은 복잡한 암호가 아니라 “어떤 글자 모양을 찾을 것인가”를 표현하는 필터입니다.

| 패턴 | 쉬운 의미 | 예시 |
|---|---|---|
| `\d+` | 숫자가 한 개 이상 | `2026`, `15` |
| `\s+` | 공백이 한 개 이상 | 탭, 여러 칸 |
| `[가-힣]+` | 완성형 한글 묶음 | `자연어` |
| `[^가-힣0-9 ]` | 한글·숫자·공백이 아닌 문자 | 특수문자 후보 |

```python
import re

text = "문의번호 A-104!!  배송이  2일 늦어요."
cleaned = re.sub(r"\s+", " ", text).strip()
print(cleaned)
```

예상 결과는 `문의번호 A-104!! 배송이 2일 늦어요.`입니다. 이 코드는 특수문자를 지우지 않고 연속 공백만 하나로 줄입니다.

## 탐색과 치환을 분리하기

바로 삭제하지 말고 먼저 몇 건이 패턴에 걸리는지 확인합니다.

```python
matches = [x for x in texts if re.search(r"https?://", x)]
print("URL 포함 문장:", len(matches))
print(matches[:3])
```

URL이 상품 정보라면 삭제하면 안 될 수 있습니다. 분석 결과를 본 뒤 제거, 마스킹, 별도 특성 추출 중 하나를 선택합니다.

## 정규화의 대표 선택

- Unicode 정규화: 겉보기는 같지만 내부 코드가 다른 문자를 통일
- 공백 정규화: 연속 공백과 탭을 일정하게 처리
- 반복 문자 정규화: `ㅋㅋㅋㅋㅋ`를 `ㅋㅋ`처럼 제한
- 대소문자 통일: 영어 검색에는 유용하지만 제품 코드에는 위험
- 숫자 치환: 모든 숫자를 `<NUM>`으로 바꾸면 패턴은 단순해지지만 금액 차이는 사라짐

```python
import unicodedata

normalized = unicodedata.normalize("NFC", text)
```

한국어 완성형 텍스트에서는 NFC가 일반적이지만 데이터 출처와 tokenizer 요구사항을 확인합니다.

## 토큰화는 문장을 모델의 단위로 나누는 일

“아버지가방에들어가신다”를 어디서 나누느냐에 따라 의미가 달라집니다. 한국어는 조사와 어미가 붙으므로 공백만으로 단어 경계를 판단하기 어렵습니다.

| 방법 | 장점 | 한계 |
|---|---|---|
| 공백 토큰화 | 빠르고 직관적 | 띄어쓰기 오류와 조사 처리 약함 |
| 형태소 분석 | 문법 단위 확인 | 도구별 결과 차이, 속도 비용 |
| Subword | 미등록어 완화 | 사람이 읽기 어려운 조각 발생 |
| 문자 단위 | 미등록어 없음 | 길이가 길고 의미 단위가 작음 |

## 한국어 전처리에서 잃기 쉬운 정보

- `안 좋아요`의 `안`을 불용어로 제거하면 감성이 반대가 됨
- `A/S`, `C++`, `100GB`의 특수문자를 지우면 제품 의미가 손상됨
- 이모지는 감성 신호일 수 있음
- 띄어쓰기 오류 자체가 사용자 집단이나 스팸의 특징일 수 있음
- 존댓말 어미는 의도·관계 표현에 도움이 될 수 있음

## 전처리 품질을 숫자로 확인하기

$$
변경률=\frac{원문과달라진문장수}{전체문장수}
$$

$$
빈문장률=\frac{전처리후빈문장수}{전체문장수}
$$

변경률이 95%라고 나쁜 것은 아니지만 예상보다 크면 규칙이 과도한지 확인합니다. 빈 문장은 모델 입력 전에 반드시 처리합니다.

```python
changed = sum(a != b for a, b in zip(raw_texts, clean_texts))
empty = sum(not x.strip() for x in clean_texts)
print(changed / len(raw_texts), empty / len(clean_texts))
```

## 전처리 전후 비교표

| 원문 | 처리 결과 | 판단 |
|---|---|---|
| `배송이  2일 늦음` | `배송이 2일 늦음` | 공백 통일 적절 |
| `좋아요!!!` | `좋아요` | 감성 강도 손실 가능 |
| `A-104 오류` | `A 오류` | 제품 코드 손상 |
| `안 만족합니다` | `만족합니다` | 부정어 제거 오류 |

최소 50~100건을 무작위로 눈으로 비교하고, 고유명사·숫자·부정·이모지 같은 중요 유형을 별도 검사합니다.
