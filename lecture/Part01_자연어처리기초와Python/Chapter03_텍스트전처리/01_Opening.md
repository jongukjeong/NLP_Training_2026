# 01. 전처리 설계 원칙

## 전처리는 모델보다 먼저 시작된다

같은 의미의 문장도 저장 방식은 다를 수 있습니다.

```text
배송이 빨라요!
배송이   빨라요
배송이 빨라요!!!
```

이 차이를 모두 제거하면 통계가 안정될 수 있지만, 느낌표가 감정 강도의 단서라면 정보 손실이 발생합니다. 따라서 전처리는 목적과 평가 기준을 먼저 정해야 합니다.

## 권장 처리 순서

```text
원본 보존 → 문자·인코딩 진단 → Unicode 정규화 → 공백 정리
→ 민감정보 마스킹 → 목적별 정제 → 토큰화 → 품질 검증 → 저장
```

## 반드시 결정할 항목

| 결정 | 예시 질문 |
|---|---|
| 분석 목적 | 검색, 감성 분류, 요약 중 무엇인가? |
| 보존 대상 | 이모지, 숫자, 문장부호가 신호인가? |
| 치환 방식 | URL을 삭제할까, `<URL>`로 바꿀까? |
| 중복 기준 | 정규화 후 같은 문장을 중복으로 볼까? |
| 검증 기준 | 제거율과 토큰 수 변화의 허용 범위는? |

## 삭제보다 마스킹

URL, 이메일, 전화번호를 완전히 삭제하면 해당 정보가 있었다는 사실까지 사라집니다. 많은 작업에서는 `<URL>`, `<EMAIL>`, `<PHONE>`처럼 유형을 보존하는 마스킹이 더 유용합니다.

## 원본과 처리 결과 함께 보기

```python
comparison = df[["text"]].copy()
comparison["clean_text"] = df["text"].map(preprocess_text)
```

무작위 표본과 가장 많이 변한 표본을 함께 검토해야 과도한 정제를 발견할 수 있습니다.

## 완료 기준

- 원본 열을 보존한다.
- 규칙과 적용 순서를 문서화한다.
- 처리 전후 빈 문장·길이·토큰 수를 비교한다.
- 민감정보가 출력과 로그에 남지 않는지 확인한다.
