# 01. 전처리 설계 원칙

# 01. 전처리 설계 원칙

## 전처리는 모델보다 먼저 시작된다

같은 의미의 문장도 저장 방식은 다를 수 있습니다.

```text
배송이 빨라요!
배송이   빨라요
배송이 빨라요!!!
```

이 차이를 모두 제거하면 통계가 안정될 수 있지만, 느낌표가 감정 강도의 단서라면 정보 손실이 발생합니다. 따라서 전처리는 목적과 평가 기준을 먼저 정해야 합니다.

## 권장 처리 순서

```text
원본 보존 → 문자·인코딩 진단 → Unicode 정규화 → 공백 정리
→ 민감정보 마스킹 → 목적별 정제 → 토큰화 → 품질 검증 → 저장
```

## 반드시 결정할 항목

| 결정 | 예시 질문 |
|---|---|
| 분석 목적 | 검색, 감성 분류, 요약 중 무엇인가? |
| 보존 대상 | 이모지, 숫자, 문장부호가 신호인가? |
| 치환 방식 | URL을 삭제할까, `<URL>`로 바꿀까? |
| 중복 기준 | 정규화 후 같은 문장을 중복으로 볼까? |
| 검증 기준 | 제거율과 토큰 수 변화의 허용 범위는? |

## 삭제보다 마스킹

URL, 이메일, 전화번호를 완전히 삭제하면 해당 정보가 있었다는 사실까지 사라집니다. 많은 작업에서는 `<URL>`, `<EMAIL>`, `<PHONE>`처럼 유형을 보존하는 마스킹이 더 유용합니다.

## 원본과 처리 결과 함께 보기

```python
comparison = df[["text"]].copy()
comparison["clean_text"] = df["text"].map(preprocess_text)
```

무작위 표본과 가장 많이 변한 표본을 함께 검토해야 과도한 정제를 발견할 수 있습니다.

## 완료 기준

- 원본 열을 보존한다.
- 규칙과 적용 순서를 문서화한다.
- 처리 전후 빈 문장·길이·토큰 수를 비교한다.
- 민감정보가 출력과 로그에 남지 않는지 확인한다.

# 02. 정규표현식 기반 정제

# 02. 정규표현식 기반 정제

## 정규표현식의 역할

정규표현식은 문자 패턴을 탐색하고 치환하는 도구입니다. Python에서는 `re` 모듈을 사용합니다.

```python
import re

text = "문의: help@example.com, https://example.com"
emails = re.findall(r"[\w.+-]+@[\w-]+(?:\.[\w-]+)+", text)
urls = re.findall(r"https?://\S+", text)
```

## 실무에서 자주 쓰는 패턴

```python
PATTERNS = {
    "url": re.compile(r"https?://\S+|www\.\S+", re.IGNORECASE),
    "email": re.compile(r"[\w.+-]+@[\w-]+(?:\.[\w-]+)+"),
    "phone": re.compile(r"(?<!\d)01[016789][ -]?\d{3,4}[ -]?\d{4}(?!\d)"),
    "spaces": re.compile(r"\s+"),
}
```

```python
def mask_patterns(text: str) -> str:
    text = PATTERNS["url"].sub("<URL>", text)
    text = PATTERNS["email"].sub("<EMAIL>", text)
    text = PATTERNS["phone"].sub("<PHONE>", text)
    return PATTERNS["spaces"].sub(" ", text).strip()
```

## 탐욕적 패턴 주의

`.*`는 가능한 한 길게 일치합니다. HTML 태그를 단순 제거할 때도 예상보다 넓은 범위가 삭제될 수 있습니다.

```python
re.sub(r"<.*?>", " ", text)  # 최소 반복이지만 완전한 HTML 파서는 아님
```

HTML 구조가 중요하면 정규표현식보다 전용 파서를 사용합니다.

## 한글과 영문을 남기는 규칙

```python
cleaned = re.sub(r"[^가-힣A-Za-z0-9\s!?.,<>]", " ", text)
```

허용 문자 방식은 이모지와 다른 언어를 제거할 수 있습니다. 적용 전에 데이터 언어와 분석 목적을 확인합니다.

## 반복 문자 완화

```python
def reduce_repeats(text: str, limit: int = 2) -> str:
    return re.sub(r"(.)\1{" + str(limit) + r",}", r"\1" * limit, text)
```

`ㅋㅋㅋㅋ`를 `ㅋㅋ`로 줄이는 것이 감성 분류에 도움이 될 수 있지만 반복 횟수 자체가 신호라면 원본 특징을 별도로 보존합니다.

## 테스트 사례

| 입력 | 기대 결과 |
|---|---|
| `https://a.com 확인` | `<URL> 확인` |
| `help@example.com` | `<EMAIL>` |
| `010-1234-5678` | `<PHONE>` |
| `정말    좋아요` | `정말 좋아요` |

# 03. Unicode와 텍스트 정규화

# 03. Unicode와 텍스트 정규화

## 눈에는 같지만 다른 문자

Unicode에서는 화면상 비슷한 문자가 서로 다른 코드 포인트 조합일 수 있습니다. 그대로 비교하면 중복 탐지와 검색이 실패할 수 있습니다.

```python
import unicodedata

normalized = unicodedata.normalize("NFC", text)
```

## NFC와 NFKC

| 방식 | 특징 | 권장 용도 |
|---|---|---|
| NFC | 표준적으로 결합 가능한 문자를 결합 | 일반 한국어 원문 보존 |
| NFKC | 호환 문자를 더 적극적으로 통합 | 검색 키·식별자 정규화 |

NFKC는 전각 문자 등을 통합하지만 표현 차이를 잃을 수 있으므로 원본 열은 보존합니다.

## 공백과 제어 문자

```python
import re
import unicodedata


def normalize_text(text: str) -> str:
    text = unicodedata.normalize("NFC", text)
    text = text.replace("\u200b", "")  # zero-width space
    text = text.replace("\xa0", " ")  # non-breaking space
    return re.sub(r"\s+", " ", text).strip()
```

## 대소문자와 숫자

영문 검색에서는 `casefold()`가 `lower()`보다 폭넓은 비교 정규화를 제공합니다. 반면 고유명사 표기나 개체명 인식에서는 대소문자를 보존해야 할 수 있습니다.

숫자를 모두 `<NUM>`으로 바꾸면 희소성은 줄지만 날짜·버전·가격 정보가 사라집니다. 목적별로 날짜, 금액, 일반 숫자를 구분해 마스킹할 수 있습니다.

## 문장부호 정책

- 감성 분석: `!`, `?`, 반복 횟수가 신호일 수 있음
- 검색: 문장부호 차이를 줄이는 편이 유리할 수 있음
- 개체명 인식: 하이픈과 점이 제품명·주소의 일부일 수 있음
- 생성 모델 학습: 원문의 문장 구조를 가능한 보존

## 정규화 기록

전처리 설정에 다음을 남깁니다.

```json
{
  "unicode_form": "NFC",
  "collapse_whitespace": true,
  "mask_url": true,
  "mask_email": true,
  "lowercase_english": false
}
```

# 04. 토큰화

# 04. 토큰화

토큰화는 텍스트를 모델이나 통계가 처리할 단위로 나누는 과정입니다. 하나의 정답이 아니라 목적과 모델에 따른 선택입니다.

## 토큰 단위 비교

| 단위 | 예 | 장점 | 한계 |
|---|---|---|---|
| 문자 | `자`, `연`, `어` | 미등록어 없음 | 시퀀스가 길어짐 |
| 어절 | `자연어`, `처리는` | 단순·빠름 | 조사·어미가 결합됨 |
| 형태소 | `자연어`, `처리`, `는` | 한국어 구조 반영 | 분석기·사전 의존 |
| 서브워드 | 모델별 조각 | 미등록어와 어휘 크기 균형 | 모델 토크나이저 필요 |

## 문장 분리

마침표만으로 분리하면 URL, 소수, 약어를 잘못 나눌 수 있습니다. 단순 예제는 한계를 명시하고 실제 시스템에서는 언어별 문장 분리기를 검토합니다.

```python
import re


def split_sentences(text: str) -> list[str]:
    parts = re.split(r"(?<=[.!?])\s+|[\r\n]+", text)
    return [part.strip() for part in parts if part.strip()]
```

## 어절 토큰화

```python
def whitespace_tokenize(text: str) -> list[str]:
    return text.split()
```

빠른 기준선에는 유용하지만 `좋다`, `좋아요`, `좋았습니다`를 서로 다른 토큰으로 봅니다.

## 모델 토크나이저

BERT나 GPT 계열 모델은 모델과 함께 제공되는 토크나이저를 사용해야 합니다. 같은 문장도 모델마다 토큰 ID가 다르며, 임의 토큰화 결과를 모델 입력에 사용하면 안 됩니다.

## 품질 지표

- 문장당 평균 토큰 수
- 빈 토큰 결과 비율
- 지나치게 긴 문장 비율
- 고유 토큰 수와 저빈도 토큰 비율
- 처리 전후 표본 비교

## 재현성

토크나이저 이름, 버전, 설정, 사용자 사전을 함께 기록합니다. 모델 토크나이저는 모델 리비전도 고정하는 것이 좋습니다.

# 05. 한국어 전처리와 형태소 분석

# 05. 한국어 전처리와 형태소 분석

## 한국어 처리의 특징

한국어는 조사와 어미가 결합하며 띄어쓰기 변이가 많습니다.

```text
배송이 빨라요
배송은 빨랐어요
배송빨라요
```

공백 토큰화만으로는 공통 의미인 `배송`, `빠르다`를 안정적으로 묶기 어렵습니다.

## 형태소 분석 결과

형태소 분석기는 일반적으로 형태소와 품사 정보를 반환합니다.

```text
배송/NNG + 이/JKS + 빠르/VA + 아요/EF
```

분석 결과는 도구, 사전, 버전에 따라 달라질 수 있습니다. 결과를 절대적 정답으로 보기보다 목적에 맞게 평가해야 합니다.

## 도구 선택 기준

| 기준 | 확인 내용 |
|---|---|
| 설치성 | Windows와 Python 3.11에서 설치 가능한가? |
| 속도 | 전체 데이터 처리 시간이 허용되는가? |
| 사전 | 도메인 용어와 사용자 사전을 지원하는가? |
| 품사 체계 | 필요한 품사를 안정적으로 구분하는가? |
| 유지보수 | 버전과 문서가 관리되고 있는가? |

교육 필수 예제는 환경 문제를 줄이기 위해 표준 라이브러리 기반 정규화와 어절 토큰화를 사용합니다. 형태소 분석기는 교육 환경에서 설치·검증한 뒤 선택 실습으로 추가합니다.

## 불용어 처리

고정 목록으로 조사·일반 단어를 무조건 삭제하면 문맥이 손실될 수 있습니다. 특히 부정어 `안`, `못`, `없다`는 감성 분류에서 중요합니다.

불용어는 다음 근거로 결정합니다.

1. 작업 목적
2. 데이터 빈도와 분포
3. 제거 전후 평가 결과
4. 도메인 전문가 검토

## 사용자 사전

제품명, 조직명, 전문 용어가 잘못 분해되면 사용자 사전이 필요합니다. 사전 파일에는 용어 출처, 추가 이유, 적용 버전을 기록합니다.

## 개인정보 마스킹

형태소 분석 전에 이메일·전화번호 등 민감정보를 마스킹하면 분석 결과나 로그에 원문이 남는 위험을 줄일 수 있습니다. 마스킹 토큰은 분석기가 분해하지 않는 형태인지 확인합니다.

# 06. 핵심 정리

## 전처리 파이프라인

```text
원본 보존 → Unicode·공백 정규화 → 패턴 마스킹
→ 목적별 문자 처리 → 토큰화 → 통계·표본 검증 → 결과 저장
```

| 주제 | 핵심 |
|---|---|
| 정규표현식 | 탐지·마스킹 규칙을 테스트 사례와 함께 관리한다. |
| Unicode | NFC/NFKC의 정보 통합 범위를 구분한다. |
| 공백 | 보이지 않는 문자와 연속 공백을 처리한다. |
| 토큰화 | 작업과 모델에 맞는 단위를 선택한다. |
| 한국어 | 조사·어미·띄어쓰기와 분석기 의존성을 고려한다. |
| 검증 | 제거율, 길이, 토큰 수와 변환 표본을 비교한다. |

## 피해야 할 전처리

- 목적 확인 없이 모든 숫자와 문장부호 삭제
- 민감정보를 로그에 출력한 뒤 마스킹
- 원본 열 또는 원본 파일 덮어쓰기
- 정제 후 빈 문장과 과도한 길이 변화를 확인하지 않기
- 형태소 분석기와 사용자 사전 버전을 기록하지 않기

## 완료 체크리스트

- [ ] 원본과 정제 텍스트가 모두 보존된다.
- [ ] URL·이메일·전화번호가 마스킹된다.
- [ ] Unicode와 공백 정책이 명시돼 있다.
- [ ] 토큰화 기준과 도구 버전을 기록한다.
- [ ] 전후 통계와 표본 검토 결과를 남긴다.

다음 장에서는 정제·토큰화한 텍스트를 BoW, TF-IDF 등 수치 벡터로 표현합니다.

# 07. 퀴즈

# 07. 퀴즈

## 문제

1. URL을 삭제하는 대신 `<URL>`로 치환하는 주된 이유는 무엇인가요?
2. 일반 한국어 원문 정규화에 우선 검토할 Unicode 형식은 무엇인가요?
3. CSV 전체 처리 전에 정규표현식 규칙을 작은 표본으로 검사해야 하는 이유는 무엇인가요?
4. 공백 토큰화가 한국어에서 갖는 대표적인 한계는 무엇인가요?
5. 모델 입력에는 왜 해당 모델과 함께 제공된 토크나이저를 사용해야 하나요?
6. 전처리 품질 검증에 포함할 지표를 두 가지 적으세요.
7. 감성 분석에서 부정어를 불용어로 제거하면 안 되는 이유는 무엇인가요?

## 정답

1. URL이 존재했다는 유형 정보를 보존하기 위해서입니다.
2. NFC입니다. NFKC는 더 적극적인 호환 문자 통합이 필요할 때 검토합니다.
3. 과도한 일치와 예상하지 못한 정보 손실을 먼저 발견하기 위해서입니다.
4. 조사·어미가 어간에 결합되고 띄어쓰기 변이를 처리하지 못합니다.
5. 모델마다 토큰 분할과 토큰 ID 체계가 다르기 때문입니다.
6. 예: 빈 문장 비율, 평균 길이, 평균 토큰 수, 마스킹 건수, 가장 많이 변한 표본.
7. `안`, `못`, `없다` 등이 감정의 방향을 바꾸는 핵심 신호이기 때문입니다.

# 08. 실습 과제

# 08. 실습 과제

## 과제: 고객 리뷰 전처리

`reviews_raw.csv`의 `text` 열을 다음 규칙으로 처리합니다.

1. 원본 `text`를 보존합니다.
2. Unicode NFC 정규화를 적용합니다.
3. zero-width space와 연속 공백을 정리합니다.
4. URL, 이메일, 휴대전화 번호를 각각 `<URL>`, `<EMAIL>`, `<PHONE>`으로 마스킹합니다.
5. 정제 결과를 `clean_text` 열에 저장합니다.
6. 공백 기준 토큰 수를 `token_count`에 저장합니다.
7. 빈 결과가 생기지 않았는지 검증합니다.
8. 결과 CSV와 처리 보고서 JSON을 `output/`에 저장합니다.

# 09. 미니 프로젝트: 전처리 품질 리포터

# 09. 미니 프로젝트: 전처리 품질 리포터

## 목표

CSV의 텍스트를 전처리하고 결과뿐 아니라 **어떤 변화가 얼마나 발생했는지** JSON 보고서로 남기는 배포 가능한 프로그램을 완성합니다.

## 필수 기능

1. 입력·출력 경로를 명령행 인수로 선택하며 기본 경로를 제공합니다.
2. `id`, `text` 필수 열을 검사합니다.
3. Unicode NFC, 보이지 않는 문자, 연속 공백을 처리합니다.
4. URL·이메일·전화번호를 유형 토큰으로 마스킹합니다.
5. 원본과 정제 텍스트, 문자 수와 토큰 수를 함께 저장합니다.
6. 처리 행 수, 마스킹 수, 평균 길이 변화, 빈 결과 수를 보고합니다.
7. 입력 파일은 덮어쓰지 않습니다.
