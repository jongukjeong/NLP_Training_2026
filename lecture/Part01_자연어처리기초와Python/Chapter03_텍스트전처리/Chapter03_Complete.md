## 핵심 용어 미리보기

본문을 읽기 전에 다음 용어의 뜻과 영어 원어를 먼저 확인합니다. 전체 정의는 [교육과정 핵심 용어집](../../TERMINOLOGY.md)에서 확인할 수 있습니다.

- **정규표현식(Regular Expression): 문자열 패턴을 찾고 바꾸기 위한 표현 규칙**
- **유니코드(Unicode): 여러 언어의 문자를 일관된 번호로 표현하는 표준**
- **정규화(Normalization): 서로 다른 문자·공백 표기를 일정한 기준으로 맞추는 처리**
- **토큰화(Tokenization): 문장을 모델이 처리할 작은 단위로 나누는 과정**

텍스트 전처리는 원문을 무조건 짧고 단순하게 만드는 작업이 아니라, **분석 목적에 맞는 정보를 보존하면서 표현의 불일치를 줄이는 과정**입니다. 이 장은 Python 기본 문법을 전제로 정규표현식, Unicode 정규화, 토큰화, 한국어 처리와 품질 검증을 다룹니다.

# Chapter 3 쉬운 전처리 판단 가이드

## 전처리는 청소가 아니라 기준 통일이다

텍스트 전처리를 “지저분한 글자를 지우는 작업”으로만 생각하면 중요한 정보까지 없앨 수 있습니다. 전처리의 진짜 목적은 같은 의미를 가능한 한 같은 형태로 표현하고, 모델이 불필요한 차이에 흔들리지 않게 만드는 것입니다.

예를 들어 감성 분석에서 `좋아요`, `좋아용`, `좋아요!!!`를 모두 비슷한 표현으로 볼 수 있습니다. 그러나 스팸 탐지에서는 반복 느낌표가 중요한 신호일 수 있습니다. 따라서 전처리는 작업 목적에 따라 달라집니다.

## 먼저 질문할 다섯 가지

1. 최종 과제는 분류, 검색, 요약 중 무엇인가?
2. 제거하려는 문자가 의미를 갖는가?
3. 대소문자·띄어쓰기 차이를 합쳐도 되는가?
4. 숫자와 고유명사를 보존해야 하는가?
5. 처리 전 원문을 다시 확인할 수 있는가?

원문 열과 정제 열을 함께 보관하면 전처리 오류를 추적하기 쉽습니다.

## 정규표현식을 필터로 이해하기

정규표현식은 복잡한 암호가 아니라 “어떤 글자 모양을 찾을 것인가”를 표현하는 필터입니다.

| 패턴 | 쉬운 의미 | 예시 |
|---|---|---|
| `\d+` | 숫자가 한 개 이상 | `2026`, `15` |
| `\s+` | 공백이 한 개 이상 | 탭, 여러 칸 |
| `[가-힣]+` | 완성형 한글 묶음 | `자연어` |
| `[^가-힣0-9 ]` | 한글·숫자·공백이 아닌 문자 | 특수문자 후보 |

```python
import re

text = "문의번호 A-104!!  배송이  2일 늦어요."
cleaned = re.sub(r"\s+", " ", text).strip()
print(cleaned)
```

예상 결과는 `문의번호 A-104!! 배송이 2일 늦어요.`입니다. 이 코드는 특수문자를 지우지 않고 연속 공백만 하나로 줄입니다.

## 탐색과 치환을 분리하기

바로 삭제하지 말고 먼저 몇 건이 패턴에 걸리는지 확인합니다.

```python
matches = [x for x in texts if re.search(r"https?://", x)]
print("URL 포함 문장:", len(matches))
print(matches[:3])
```

URL이 상품 정보라면 삭제하면 안 될 수 있습니다. 분석 결과를 본 뒤 제거, 마스킹, 별도 특성 추출 중 하나를 선택합니다.

## 정규화의 대표 선택

- Unicode 정규화: 겉보기는 같지만 내부 코드가 다른 문자를 통일
- 공백 정규화: 연속 공백과 탭을 일정하게 처리
- 반복 문자 정규화: `ㅋㅋㅋㅋㅋ`를 `ㅋㅋ`처럼 제한
- 대소문자 통일: 영어 검색에는 유용하지만 제품 코드에는 위험
- 숫자 치환: 모든 숫자를 `<NUM>`으로 바꾸면 패턴은 단순해지지만 금액 차이는 사라짐

```python
import unicodedata

normalized = unicodedata.normalize("NFC", text)
```

한국어 완성형 텍스트에서는 NFC가 일반적이지만 데이터 출처와 tokenizer 요구사항을 확인합니다.

## 토큰화는 문장을 모델의 단위로 나누는 일

“아버지가방에들어가신다”를 어디서 나누느냐에 따라 의미가 달라집니다. 한국어는 조사와 어미가 붙으므로 공백만으로 단어 경계를 판단하기 어렵습니다.

| 방법 | 장점 | 한계 |
|---|---|---|
| 공백 토큰화 | 빠르고 직관적 | 띄어쓰기 오류와 조사 처리 약함 |
| 형태소 분석 | 문법 단위 확인 | 도구별 결과 차이, 속도 비용 |
| Subword | 미등록어 완화 | 사람이 읽기 어려운 조각 발생 |
| 문자 단위 | 미등록어 없음 | 길이가 길고 의미 단위가 작음 |

## 한국어 전처리에서 잃기 쉬운 정보

- `안 좋아요`의 `안`을 불용어로 제거하면 감성이 반대가 됨
- `A/S`, `C++`, `100GB`의 특수문자를 지우면 제품 의미가 손상됨
- 이모지는 감성 신호일 수 있음
- 띄어쓰기 오류 자체가 사용자 집단이나 스팸의 특징일 수 있음
- 존댓말 어미는 의도·관계 표현에 도움이 될 수 있음

## 전처리 품질을 숫자로 확인하기

\[
변경률=\frac{원문과달라진문장수}{전체문장수}
\]

\[
빈문장률=\frac{전처리후빈문장수}{전체문장수}
\]

변경률이 95%라고 나쁜 것은 아니지만 예상보다 크면 규칙이 과도한지 확인합니다. 빈 문장은 모델 입력 전에 반드시 처리합니다.

```python
changed = sum(a != b for a, b in zip(raw_texts, clean_texts))
empty = sum(not x.strip() for x in clean_texts)
print(changed / len(raw_texts), empty / len(clean_texts))
```

## 전처리 전후 비교표

| 원문 | 처리 결과 | 판단 |
|---|---|---|
| `배송이  2일 늦음` | `배송이 2일 늦음` | 공백 통일 적절 |
| `좋아요!!!` | `좋아요` | 감성 강도 손실 가능 |
| `A-104 오류` | `A 오류` | 제품 코드 손상 |
| `안 만족합니다` | `만족합니다` | 부정어 제거 오류 |

최소 50~100건을 무작위로 눈으로 비교하고, 고유명사·숫자·부정·이모지 같은 중요 유형을 별도 검사합니다.

## 재현 가능한 전처리

규칙을 노트북 셀에 흩어 두지 말고 함수와 설정으로 관리합니다. 전처리 버전, 사전 파일, 형태소 분석기 버전을 기록합니다. 학습과 서비스에서 반드시 같은 함수를 사용합니다.

# Chapter 3 전처리 실험과 오류 분석

## 전처리 규칙을 실험으로 결정하기

전처리 규칙은 “깔끔해 보인다”가 아니라 최종 과제 성능과 오류 유형으로 선택합니다. 규칙 A와 B를 같은 데이터 분할에서 비교하고 한 번에 한 규칙만 바꿉니다.

| 실험 | 변경 규칙 | 빈 문장률 | 미등록어율 | Val F1 | 결론 |
|---|---|---:|---:|---:|---|
| base | 공백만 통일 | 기록 | 기록 | 기록 | 기준 |
| exp01 | 특수문자 제거 | 기록 | 기록 | 기록 | 비교 |
| exp02 | 반복문자 2회 제한 | 기록 | 기록 | 기록 | 비교 |

## 처리 함수의 입출력 계약

```python
def preprocess(text: str) -> str:
    """문자열 하나를 받아 정제 문자열 하나를 반환한다."""
    if text is None:
        return ""
    text = str(text)
    text = normalize_unicode(text)
    text = normalize_spaces(text)
    return text.strip()
```

함수는 같은 입력에 같은 결과를 내야 하며 학습과 서비스에서 공유합니다. 데이터프레임 전체에 의존하는 통계 처리는 별도 `fit/transform` 단계로 구분합니다.

## 규칙별 단위 테스트

```python
cases = [
    ("배송이  늦어요", "배송이 늦어요"),
    ("  환불 요청  ", "환불 요청"),
    ("A-104 오류", "A-104 오류"),
]
for raw, expected in cases:
    assert preprocess(raw) == expected
```

대표 사례뿐 아니라 빈 문자열, `None`, 숫자만 있는 문장, 이모지, 매우 긴 문장을 포함합니다.

## 과도한 정제 찾기

원문과 결과 길이 비율을 계산합니다.

\[
보존율=\frac{전처리후문자수}{원본문자수}
\]

보존율이 0.1인 문장은 원문의 90%가 사라졌다는 뜻입니다. 낮은 보존율 사례를 정렬해 규칙을 검토합니다.

```python
ratio = len(cleaned) / max(len(raw), 1)
```

## 부정어 보존 검사

`안`, `못`, `없다`, `아니다`는 감성을 바꿉니다. 전처리 후 이 표현이 사라진 문장을 자동으로 찾습니다.

```python
negations = ["안", "못", "없", "아니"]
if any(n in raw for n in negations) and not any(n in clean for n in negations):
    print("부정 정보 손실", raw, clean)
```

## 숫자와 코드

검색·고객지원에서는 `A-104`, `5G`, `100GB`, 날짜와 금액이 핵심일 수 있습니다. 숫자를 모두 `<NUM>`으로 바꾸는 실험과 원문 유지 실험을 분리합니다.

## 형태소 분석 결과 검토

형태소 분석기는 정답 생성기가 아닙니다. 도메인 신조어와 제품명을 잘못 나눌 수 있습니다.

```text
원문: 챗GPT플러스 결제 오류
도구 A: 챗 / GPT / 플러스 / 결제 / 오류
도구 B: 챗GPT플러스 / 결제 / 오류
```

최종 과제에서 어떤 결과가 더 유용한지 판단하고 사용자 사전 적용 여부를 비교합니다.

## Tokenizer 변경의 영향

Tokenizer를 바꾸면 vocabulary, sequence length, OOV와 모델 입력이 모두 변합니다. 이전 모델 가중치나 vectorizer와 섞어 쓰지 않습니다.

## 데이터 분할 시점

중복 제거와 그룹 분할을 먼저 계획합니다. 전체 데이터 빈도를 보고 불용어를 선택하면 test 정보가 들어갈 수 있습니다. 학습 데이터로 규칙을 결정하고 test는 마지막 확인에 사용합니다.

## 오류 샘플링

무작위 50건 외에 다음 위험 그룹을 각각 20건씩 확인합니다.

- 가장 긴 문장
- 가장 많이 변경된 문장
- 숫자·URL·이모지 포함
- 부정어 포함
- 처리 후 빈 문자열
- 희귀 문자와 외국어 포함

## 실습 보고서 문장

“특수문자를 제거했다”보다 “전체 2,000건 중 41%가 변경됐고 12건이 빈 문자열이 됐다. 제품 코드 손실 7건을 확인해 하이픈은 보존하도록 규칙을 수정했다”고 결과와 판단 근거를 작성합니다.

## 완료 기준

1. 규칙별 단위 테스트 존재
2. 원문·정제문 동시 보관
3. 변경률·빈 문장률·보존율 측정
4. 중요 표현 손실 검사
5. 학습·서비스 함수 일치
6. 전처리 버전 기록

# 01. 전처리 설계 원칙

## 전처리는 모델보다 먼저 시작된다

같은 의미의 문장도 저장 방식은 다를 수 있습니다.

```text
배송이 빨라요!
배송이   빨라요
배송이 빨라요!!!
```

이 차이를 모두 제거하면 통계가 안정될 수 있지만, 느낌표가 감정 강도의 단서라면 정보 손실이 발생합니다. 따라서 전처리는 목적과 평가 기준을 먼저 정해야 합니다.

## 권장 처리 순서

```text
원본 보존 → 문자·인코딩 진단 → Unicode 정규화 → 공백 정리
→ 민감정보 마스킹 → 목적별 정제 → 토큰화 → 품질 검증 → 저장
```

## 반드시 결정할 항목

| 결정 | 예시 질문 |
|---|---|
| 분석 목적 | 검색, 감성 분류, 요약 중 무엇인가? |
| 보존 대상 | 이모지, 숫자, 문장부호가 신호인가? |
| 치환 방식 | URL을 삭제할까, `<URL>`로 바꿀까? |
| 중복 기준 | 정규화 후 같은 문장을 중복으로 볼까? |
| 검증 기준 | 제거율과 토큰 수 변화의 허용 범위는? |

## 삭제보다 마스킹

URL, 이메일, 전화번호를 완전히 삭제하면 해당 정보가 있었다는 사실까지 사라집니다. 많은 작업에서는 `<URL>`, `<EMAIL>`, `<PHONE>`처럼 유형을 보존하는 마스킹이 더 유용합니다.

## 원본과 처리 결과 함께 보기

```python
comparison = df[["text"]].copy()
comparison["clean_text"] = df["text"].map(preprocess_text)
```

무작위 표본과 가장 많이 변한 표본을 함께 검토해야 과도한 정제를 발견할 수 있습니다.

## 완료 기준

- 원본 열을 보존한다.
- 규칙과 적용 순서를 문서화한다.
- 처리 전후 빈 문장·길이·토큰 수를 비교한다.
- 민감정보가 출력과 로그에 남지 않는지 확인한다.

# 02. 정규표현식 기반 정제

## 정규표현식의 역할

정규표현식은 문자 패턴을 탐색하고 치환하는 도구입니다. Python에서는 `re` 모듈을 사용합니다.

```python
import re

text = "문의: help@example.com, https://example.com"
emails = re.findall(r"[\w.+-]+@[\w-]+(?:\.[\w-]+)+", text)
urls = re.findall(r"https?://\S+", text)
```

## 실무에서 자주 쓰는 패턴

```python
PATTERNS = {
    "url": re.compile(r"https?://\S+|www\.\S+", re.IGNORECASE),
    "email": re.compile(r"[\w.+-]+@[\w-]+(?:\.[\w-]+)+"),
    "phone": re.compile(r"(?<!\d)01[016789][ -]?\d{3,4}[ -]?\d{4}(?!\d)"),
    "spaces": re.compile(r"\s+"),
}
```

```python
def mask_patterns(text: str) -> str:
    text = PATTERNS["url"].sub("<URL>", text)
    text = PATTERNS["email"].sub("<EMAIL>", text)
    text = PATTERNS["phone"].sub("<PHONE>", text)
    return PATTERNS["spaces"].sub(" ", text).strip()
```

## 탐욕적 패턴 주의

`.*`는 가능한 한 길게 일치합니다. HTML 태그를 단순 제거할 때도 예상보다 넓은 범위가 삭제될 수 있습니다.

```python
re.sub(r"<.*?>", " ", text)  # 최소 반복이지만 완전한 HTML 파서는 아님
```

HTML 구조가 중요하면 정규표현식보다 전용 파서를 사용합니다.

## 한글과 영문을 남기는 규칙

```python
cleaned = re.sub(r"[^가-힣A-Za-z0-9\s!?.,<>]", " ", text)
```

허용 문자 방식은 이모지와 다른 언어를 제거할 수 있습니다. 적용 전에 데이터 언어와 분석 목적을 확인합니다.

## 반복 문자 완화

```python
def reduce_repeats(text: str, limit: int = 2) -> str:
    return re.sub(r"(.)\1{" + str(limit) + r",}", r"\1" * limit, text)
```

`ㅋㅋㅋㅋ`를 `ㅋㅋ`로 줄이는 것이 감성 분류에 도움이 될 수 있지만 반복 횟수 자체가 신호라면 원본 특징을 별도로 보존합니다.

## 테스트 사례

| 입력 | 기대 결과 |
|---|---|
| `https://a.com 확인` | `<URL> 확인` |
| `help@example.com` | `<EMAIL>` |
| `010-1234-5678` | `<PHONE>` |
| `정말    좋아요` | `정말 좋아요` |

# 03. Unicode와 텍스트 정규화

## 눈에는 같지만 다른 문자

Unicode에서는 화면상 비슷한 문자가 서로 다른 코드 포인트 조합일 수 있습니다. 그대로 비교하면 중복 탐지와 검색이 실패할 수 있습니다.

```python
import unicodedata

normalized = unicodedata.normalize("NFC", text)
```

## NFC와 NFKC

| 방식 | 특징 | 권장 용도 |
|---|---|---|
| NFC | 표준적으로 결합 가능한 문자를 결합 | 일반 한국어 원문 보존 |
| NFKC | 호환 문자를 더 적극적으로 통합 | 검색 키·식별자 정규화 |

NFKC는 전각 문자 등을 통합하지만 표현 차이를 잃을 수 있으므로 원본 열은 보존합니다.

## 공백과 제어 문자

```python
import re
import unicodedata

def normalize_text(text: str) -> str:
    text = unicodedata.normalize("NFC", text)
    text = text.replace("\u200b", "")  # zero-width space
    text = text.replace("\xa0", " ")  # non-breaking space
    return re.sub(r"\s+", " ", text).strip()
```

## 대소문자와 숫자

영문 검색에서는 `casefold()`가 `lower()`보다 폭넓은 비교 정규화를 제공합니다. 반면 고유명사 표기나 개체명 인식에서는 대소문자를 보존해야 할 수 있습니다.

숫자를 모두 `<NUM>`으로 바꾸면 희소성은 줄지만 날짜·버전·가격 정보가 사라집니다. 목적별로 날짜, 금액, 일반 숫자를 구분해 마스킹할 수 있습니다.

## 문장부호 정책

- 감성 분석: `!`, `?`, 반복 횟수가 신호일 수 있음
- 검색: 문장부호 차이를 줄이는 편이 유리할 수 있음
- 개체명 인식: 하이픈과 점이 제품명·주소의 일부일 수 있음
- 생성 모델 학습: 원문의 문장 구조를 가능한 보존

## 정규화 기록

전처리 설정에 다음을 남깁니다.

```json
{
  "unicode_form": "NFC",
  "collapse_whitespace": true,
  "mask_url": true,
  "mask_email": true,
  "lowercase_english": false
}
```

# 04. 토큰화

토큰화는 텍스트를 모델이나 통계가 처리할 단위로 나누는 과정입니다. 하나의 정답이 아니라 목적과 모델에 따른 선택입니다.

## 토큰 단위 비교

| 단위 | 예 | 장점 | 한계 |
|---|---|---|---|
| 문자 | `자`, `연`, `어` | 미등록어 없음 | 시퀀스가 길어짐 |
| 어절 | `자연어`, `처리는` | 단순·빠름 | 조사·어미가 결합됨 |
| 형태소 | `자연어`, `처리`, `는` | 한국어 구조 반영 | 분석기·사전 의존 |
| 서브워드 | 모델별 조각 | 미등록어와 어휘 크기 균형 | 모델 토크나이저 필요 |

## 문장 분리

마침표만으로 분리하면 URL, 소수, 약어를 잘못 나눌 수 있습니다. 단순 예제는 한계를 명시하고 실제 시스템에서는 언어별 문장 분리기를 검토합니다.

```python
import re

def split_sentences(text: str) -> list[str]:
    parts = re.split(r"(?<=[.!?])\s+|[\r\n]+", text)
    return [part.strip() for part in parts if part.strip()]
```

## 어절 토큰화

```python
def whitespace_tokenize(text: str) -> list[str]:
    return text.split()
```

빠른 기준선에는 유용하지만 `좋다`, `좋아요`, `좋았습니다`를 서로 다른 토큰으로 봅니다.

## 모델 토크나이저

BERT나 GPT 계열 모델은 모델과 함께 제공되는 토크나이저를 사용해야 합니다. 같은 문장도 모델마다 토큰 ID가 다르며, 임의 토큰화 결과를 모델 입력에 사용하면 안 됩니다.

## 품질 지표

- 문장당 평균 토큰 수
- 빈 토큰 결과 비율
- 지나치게 긴 문장 비율
- 고유 토큰 수와 저빈도 토큰 비율
- 처리 전후 표본 비교

## 재현성

토크나이저 이름, 버전, 설정, 사용자 사전을 함께 기록합니다. 모델 토크나이저는 모델 리비전도 고정하는 것이 좋습니다.

# 05. 한국어 전처리와 형태소 분석

## 한국어 처리의 특징

한국어는 조사와 어미가 결합하며 띄어쓰기 변이가 많습니다.

```text
배송이 빨라요
배송은 빨랐어요
배송빨라요
```

공백 토큰화만으로는 공통 의미인 `배송`, `빠르다`를 안정적으로 묶기 어렵습니다.

## 형태소 분석 결과

형태소 분석기는 일반적으로 형태소와 품사 정보를 반환합니다.

```text
배송/NNG + 이/JKS + 빠르/VA + 아요/EF
```

분석 결과는 도구, 사전, 버전에 따라 달라질 수 있습니다. 결과를 절대적 정답으로 보기보다 목적에 맞게 평가해야 합니다.

## 도구 선택 기준

| 기준 | 확인 내용 |
|---|---|
| 설치성 | Windows와 Python 3.11에서 설치 가능한가? |
| 속도 | 전체 데이터 처리 시간이 허용되는가? |
| 사전 | 도메인 용어와 사용자 사전을 지원하는가? |
| 품사 체계 | 필요한 품사를 안정적으로 구분하는가? |
| 유지보수 | 버전과 문서가 관리되고 있는가? |

교육 필수 예제는 환경 문제를 줄이기 위해 표준 라이브러리 기반 정규화와 어절 토큰화를 사용합니다. 형태소 분석기는 교육 환경에서 설치·검증한 뒤 선택 실습으로 추가합니다.

## 불용어 처리

고정 목록으로 조사·일반 단어를 무조건 삭제하면 문맥이 손실될 수 있습니다. 특히 부정어 `안`, `못`, `없다`는 감성 분류에서 중요합니다.

불용어는 다음 근거로 결정합니다.

1. 작업 목적
2. 데이터 빈도와 분포
3. 제거 전후 평가 결과
4. 도메인 전문가 검토

## 사용자 사전

제품명, 조직명, 전문 용어가 잘못 분해되면 사용자 사전이 필요합니다. 사전 파일에는 용어 출처, 추가 이유, 적용 버전을 기록합니다.

## 개인정보 마스킹

형태소 분석 전에 이메일·전화번호 등 민감정보를 마스킹하면 분석 결과나 로그에 원문이 남는 위험을 줄일 수 있습니다. 마스킹 토큰은 분석기가 분해하지 않는 형태인지 확인합니다.

# 07. 퀴즈

## 문제

1. URL을 삭제하는 대신 `<URL>`로 치환하는 주된 이유는 무엇인가요?
2. 일반 한국어 원문 정규화에 우선 검토할 Unicode 형식은 무엇인가요?
3. CSV 전체 처리 전에 정규표현식 규칙을 작은 표본으로 검사해야 하는 이유는 무엇인가요?
4. 공백 토큰화가 한국어에서 갖는 대표적인 한계는 무엇인가요?
5. 모델 입력에는 왜 해당 모델과 함께 제공된 토크나이저를 사용해야 하나요?
6. 전처리 품질 검증에 포함할 지표를 두 가지 적으세요.
7. 감성 분석에서 부정어를 불용어로 제거하면 안 되는 이유는 무엇인가요?

## 정답

1. URL이 존재했다는 유형 정보를 보존하기 위해서입니다.
2. NFC입니다. NFKC는 더 적극적인 호환 문자 통합이 필요할 때 검토합니다.
3. 과도한 일치와 예상하지 못한 정보 손실을 먼저 발견하기 위해서입니다.
4. 조사·어미가 어간에 결합되고 띄어쓰기 변이를 처리하지 못합니다.
5. 모델마다 토큰 분할과 토큰 ID 체계가 다르기 때문입니다.
6. 예: 빈 문장 비율, 평균 길이, 평균 토큰 수, 마스킹 건수, 가장 많이 변한 표본.
7. `안`, `못`, `없다` 등이 감정의 방향을 바꾸는 핵심 신호이기 때문입니다.

# 08. 실습 과제

## 과제: 고객 리뷰 전처리

`reviews_raw.csv`의 `text` 열을 다음 규칙으로 처리합니다.

1. 원본 `text`를 보존합니다.
2. Unicode NFC 정규화를 적용합니다.
3. zero-width space와 연속 공백을 정리합니다.
4. URL, 이메일, 휴대전화 번호를 각각 `<URL>`, `<EMAIL>`, `<PHONE>`으로 마스킹합니다.
5. 정제 결과를 `clean_text` 열에 저장합니다.
6. 공백 기준 토큰 수를 `token_count`에 저장합니다.
7. 빈 결과가 생기지 않았는지 검증합니다.
8. 결과 CSV와 처리 보고서 JSON을 `output/`에 저장합니다.
