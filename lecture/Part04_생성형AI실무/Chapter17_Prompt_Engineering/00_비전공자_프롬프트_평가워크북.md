# Chapter 17 비전공자용 프롬프트 평가 워크북

## 프롬프트는 좋은 문장보다 재현 가능한 작업 명세다

프롬프트 엔지니어링의 목표는 한 번 멋진 답을 얻는 것이 아닙니다. 입력이 달라져도 요구한 형식과 품질을 안정적으로 지키게 만드는 것이 목표입니다. 따라서 프롬프트를 프로그램 코드처럼 버전으로 관리하고 고정된 평가 질문으로 비교합니다.

## 기본 구성 다섯 가지

```text
역할: 어떤 관점에서 수행하는가
목표: 무엇을 완료해야 하는가
입력: 어떤 데이터가 주어지는가
제약: 무엇을 하면 안 되는가
출력 계약: 어떤 형식과 항목으로 답하는가
```

역할만 길게 쓰기보다 목표와 성공 기준을 구체적으로 적는 편이 중요합니다.

## Zero-shot과 Few-shot의 차이

Zero-shot은 설명만 주고 작업을 요청합니다. Few-shot은 입력과 기대 출력 예시를 함께 제공합니다. 예시는 단순한 정답 모음이 아니라 모델이 따라야 할 결정 경계를 보여 줍니다.

```text
입력: 배송이 아직 안 왔어요
출력: {"label":"배송"}

입력: 주문을 취소하고 싶어요
출력: {"label":"환불"}
```

예시가 실제 입력을 그대로 포함하면 평가 누수가 생길 수 있으므로 별도의 평가셋을 유지합니다.

## 정확도와 형식 준수율을 분리한다

평가 질문이 20개이고 정답이 17개라면 정확도는 다음과 같습니다.

\[
Accuracy=\frac{17}{20}=0.85
\]

이 중 JSON Schema를 지킨 답이 18개라면 형식 준수율은 90%입니다. 내용은 맞지만 형식이 틀린 경우와 형식은 맞지만 내용이 틀린 경우를 분리해야 개선 지점을 찾을 수 있습니다.

## 평가표를 먼저 만든다

| 항목 | 판정 예시 |
|---|---|
| 정답성 | 기대 Label과 같은가 |
| 형식 | JSON Schema를 통과하는가 |
| 근거성 | 입력에 없는 사실을 추가하지 않았는가 |
| 안전성 | 민감정보와 금지 행동을 피했는가 |
| 비용 | 평균 입력·출력 토큰 수 |
| 지연 | 평균과 상위 95% 응답 시간 |

## Structured Output도 프로그램에서 검사한다

```python
allowed = {"배송", "환불", "결제", "기타"}
assert set(result) == {"label", "reason"}
assert result["label"] in allowed
assert isinstance(result["reason"], str)
```

JSON처럼 보이는 문자열과 검증된 데이터는 다릅니다. Parsing, 필수 Key, 자료형, 허용 값 검사를 통과해야 다음 업무에 전달합니다.

## 내부 사고 과정 대신 검증 가능한 결과를 요청한다

긴 Chain of Thought 전체를 출력하게 하기보다 최종 답, 짧은 근거, 사용한 입력 항목과 불확실성을 요청합니다. 사용자는 결과를 확인할 수 있어야 하며 모델 내부의 상세 추론을 신뢰 근거로 삼으면 안 됩니다.

## 프롬프트 변경은 한 번에 하나씩 비교한다

1. 기준 프롬프트와 평가셋을 고정합니다.
2. 출력 형식 또는 예시처럼 한 요소만 바꿉니다.
3. 정확도·형식·비용·지연을 다시 측정합니다.
4. 좋아진 사례와 나빠진 사례를 모두 기록합니다.
5. 기준 미달이면 이전 버전으로 되돌립니다.

## Prompt Injection을 데이터로 취급한다

문서나 사용자 입력 안의 “이전 지시를 무시하라”는 문장은 신뢰할 수 없는 데이터입니다. 시스템 규칙과 사용자 데이터를 구분하고, 도구 실행·외부 전송·삭제 같은 행동은 별도 권한 검사와 사용자 확인을 거칩니다.
