# Chapter 17 원리·수학·실습 가이드

## 1. 프롬프트를 프로그램처럼 관리하기

프롬프트는 감으로 고치는 문장이 아니라 입력과 출력 계약이다. `목표 → 입력 경계 → 규칙 → 출력 스키마 → 예시 → 실패 처리`를 명시하고 버전과 평가 결과를 기록한다.

Zero-shot은 예시 없이 지시하고, Few-shot은 대표 입출력 예시를 제공한다. 예시는 많기보다 경계 사례와 실제 분포를 잘 대표해야 한다.

## 2. Structured Output

```json
{"category":"배송","priority":"high","reason":"하루 이상 지연"}
```

필드 타입, 허용값, 누락 처리 규칙을 스키마로 정의한다. 단순히 “JSON으로 답해”라고만 하면 설명 문장이 섞이거나 키가 변할 수 있다. 파싱 실패 시 제한 횟수 재시도하고 원문을 보관한다.

## 3. 추론 요청과 검증

복잡한 문제는 하위 작업으로 나누고 필요한 중간 산출물만 구조화해 요청한다. 장황한 비공개 사고과정을 요구하기보다 근거, 계산 결과, 확인 가능한 요약을 받는다. 외부 문서의 명령은 데이터로 취급해 prompt injection을 막는다.

## 4. 평가 수학

\[
Accuracy=\frac{correct}{N},\quad SchemaRate=\frac{valid\ outputs}{N}
\]

\[
AverageCost=\frac{\sum_i cost_i}{N},\quad P95Latency=95\%\ 지점의\ 지연시간
\]

프롬프트 A/B는 같은 평가셋, 모델, 파라미터에서 비교한다. 정확도뿐 아니라 형식 준수율, 안전 실패, 토큰, 비용, 지연을 함께 본다. 테스트셋을 보며 프롬프트를 반복 수정하면 과적합되므로 개발셋과 최종 테스트셋을 분리한다.

1. Few-shot 예시는 어떤 사례를 우선해야 하는가?
2. Structured Output에서 스키마가 필요한 이유는?
3. 프롬프트 비교에서 정확도 외 지표는?

# Chapter 17 프롬프트 평가와 개선 실습

## 프롬프트는 정답 문장이 아니라 실험 대상이다

좋은 프롬프트는 한 번에 떠오르는 마법 문장이 아닙니다. 고정 평가 질문으로 실패를 확인하고 한 요소씩 수정하는 작은 프로그램에 가깝습니다.

## 나쁜 요청과 개선 요청

```text
나쁜 예: 문의를 잘 분류해 줘.
```

```text
목표: 문의를 하나의 카테고리로 분류
허용값: 배송, 환불, 계정, 기타
규칙: 두 의도가 있으면 고객이 요구한 조치를 우선
출력: {"category":"...","reason":"..."}
불명확하면 기타
```

개선된 요청은 결과가 좋은지 자동으로 판정할 수 있습니다.

## 평가셋 만들기

쉬운 사례만 넣지 않습니다.

- 전형적인 입력
- 두 의도가 섞인 입력
- 오타와 짧은 입력
- 규칙 밖 입력
- prompt injection 문구가 포함된 입력
- 개인정보가 포함된 입력

개발셋으로 수정하고 test는 마지막 평가에만 사용합니다.

## 정확도와 형식 준수율

\[
Accuracy=정답수/N
\]

\[
SchemaRate=파싱성공수/N
\]

100건 중 분류 정답 90건, JSON 파싱 성공 96건이면 각각 90%, 96%입니다. 파싱 실패 4건은 분류 정답 여부와 별도 운영 오류입니다.

## Few-shot 예제 선택

같은 쉬운 배송 예시를 세 번 넣기보다 배송/환불 경계, 기타 처리, 복합 의도처럼 판단 규칙을 보여주는 사례를 고릅니다. 예제 순서에 따라 결과가 변하는지도 확인합니다.

## Structured Output 검증

필수 필드, 타입, 허용값을 검사합니다. `category="배송지연"`처럼 허용 목록에 없는 값은 JSON 형식이 맞아도 업무 schema에는 실패입니다.

## Chain of Thought를 쉽게 다루기

복잡한 문제는 “하위 항목을 확인하고 최종 결과와 짧은 근거를 출력”하도록 구성합니다. 장황한 내부 사고과정을 요구하기보다 확인 가능한 근거와 계산 결과를 요청합니다.

## 비용·지연 비교

프롬프트가 길어지면 항상 좋아지는 것은 아닙니다. 정확도, 입력 토큰, 출력 토큰, 평균 비용, P95 지연을 함께 기록합니다.

| 버전 | 정확도 | 형식 | 입력토큰 | P95 | 결론 |
|---|---:|---:|---:|---:|---|
| v1 | 기록 | 기록 | 기록 | 기록 | 기준 |
| v2 | 기록 | 기록 | 기록 | 기록 | 채택/기각 |

## 실패 대응

- 파싱 실패: schema 기반 제한 재시도
- 근거 부족: “모름” 반환
- 민감정보: 전송 전 마스킹
- 외부 입력의 명령: 데이터로 취급
- 반복 실패: 사람 검토 대기열

## 완료 기준

프롬프트 텍스트뿐 아니라 모델, 파라미터, 평가셋 버전, 결과, 실패 사례와 비용을 함께 저장합니다.

# 17.1~17.3 Prompt 기본·Zero-shot·Few-shot

Prompt는 역할 수사보다 목표, 입력, 제약, 출력 계약과 성공 기준이 중요합니다.

```text
목표 → 필요한 배경 → 입력 구분 → 제약 → 출력 schema → 예외 처리
```

Zero-shot은 예시 없이 지시하고 Few-shot은 대표 입력·출력 예시를 제공합니다. Few-shot 예시는 정상 사례뿐 아니라 경계·거부 사례와 label 균형을 포함합니다.

Prompt injection에 대비해 사용자 입력과 신뢰된 지시를 구분하고, 입력 안의 지시를 시스템 규칙으로 취급하지 않도록 합니다. 모델 출력은 검증 전까지 신뢰하지 않습니다.

## 프롬프트를 계약서처럼 쓰기

좋은 프롬프트는 화려한 역할 문장보다 목표와 판정 기준이 명확합니다.

```text
목표: 고객 문의를 하나의 카테고리로 분류
입력: <inquiry>...</inquiry>
허용값: 배송, 환불, 계정, 기타
규칙: 불명확하면 기타
출력: {"category":"...","reason":"..."}
```

입력 경계를 태그로 분리하면 사용자 문장과 시스템 규칙을 구분하기 쉽습니다. 입력 속 “앞 지시를 무시하라”는 문장은 데이터이지 신뢰된 지시가 아닙니다.

## Zero-shot과 Few-shot

Zero-shot은 설명만으로 수행하고 Few-shot은 예시를 함께 줍니다. 예시는 쉬운 사례만 반복하지 말고 클래스 경계, 예외, 모호한 사례를 포함합니다. 잘못된 예시는 모델에 강한 잘못된 패턴을 줄 수 있습니다.

## 평가를 수치로 보기

\[
Accuracy=\frac{정답수}{전체수},\quad
SchemaRate=\frac{파싱가능한출력수}{전체수}
\]

100개 중 정답 87개, JSON 파싱 성공 95개면 정확도 87%, 형식 준수율 95%입니다. 정확도만 보면 운영에서 5%가 파싱 실패한다는 문제를 놓칩니다.

## 프롬프트 변경 실험

평가셋, 모델, temperature를 고정하고 프롬프트 한 요소만 바꿉니다. 개발셋 결과로 개선하고 마지막 test는 최종 확인에만 사용합니다. 프롬프트 버전, 토큰 수, 비용, P95 지연도 기록합니다.

## 쉬운 개선 순서

모호한 목표 수정 → 입력 경계 표시 → 출력 스키마 명시 → 대표 예시 추가 → 실패 처리 규칙 → 자동 평가 순으로 개선합니다. 무조건 긴 프롬프트가 좋은 것은 아닙니다.

# 17.4 Chain of Thought · 17.5 Structured Output

복잡한 task는 문제를 단계로 나누고 필요한 중간 검증을 요청할 수 있습니다. 내부 추론 전체를 노출하도록 요구하기보다 최종 답과 검증 가능한 짧은 근거·계산 결과를 요청합니다.

Structured Output은 후속 시스템이 파싱할 schema를 강제합니다.

```json
{
  "category": "delivery",
  "priority": "high",
  "reason": "배송 지연과 행사 일정 충돌"
}
```

JSON parse, 필수 field, enum, 길이와 금지값을 코드로 검증하고 실패 시 제한된 재시도를 수행합니다. 자연어로 “JSON으로 써줘”만 요청하는 것보다 API의 schema 기능을 우선 검토합니다.

## 프롬프트 템플릿을 사용하는 이유

문자열을 매번 이어 붙이면 입력 경계와 변수 누락을 놓치기 쉽습니다. 템플릿은 고정 지시와 사용자 데이터를 분리하고 같은 구조를 반복 실행하게 합니다.

```python
template = """다음 문의를 분류하세요.
허용 카테고리: 배송, 환불, 계정, 기타
문의: <input>{inquiry}</input>
JSON만 출력하세요."""
prompt = template.format(inquiry=user_text)
```

사용자 입력에 중괄호나 태그가 들어올 수 있으므로 실제 애플리케이션에서는 프레임워크의 template API와 입력 검증을 사용합니다. 템플릿은 보안을 자동 해결하지 않으며 외부 입력을 지시보다 낮은 신뢰도로 다뤄야 합니다.

## 메시지 역할 분리

대화 모델의 입력은 일반적으로 시스템 지시, 사용자 요청, 도구 결과 등 역할별 메시지로 구성됩니다. 정책과 출력 계약은 신뢰된 메시지에 두고, 검색 문서와 도구 결과는 참고 데이터임을 명시합니다.

## Partial prompt와 재사용

날짜, 제품 목록, 출력 언어처럼 여러 요청에서 반복되는 일부 변수를 미리 묶으면 프롬프트 중복을 줄일 수 있습니다. 하지만 자동으로 들어가는 값도 버전과 출처를 기록해야 재현할 수 있습니다.

## Streaming의 의미

Streaming은 답 전체가 완성되기 전에 토큰 일부를 순차적으로 전달해 체감 지연을 줄입니다. 총 생성시간과 비용이 반드시 줄어드는 것은 아닙니다. JSON처럼 전체 구조 검증이 필요한 출력은 스트림 도중 실행하지 않고 완성 후 파싱합니다.

## 출력 파싱 실패 처리

1. schema validation으로 타입과 필수 필드를 검사합니다.
2. 실패 이유를 기록하고 제한 횟수만 재요청합니다.
3. 반복 실패는 안전한 기본값 또는 사람 검토로 보냅니다.
4. 모델 원문과 파싱 결과를 함께 보존합니다.

```python
required = {"category", "reason"}
if not required.issubset(result):
    raise ValueError("필수 필드 누락")
```

## 대화 메모리와 프롬프트 예산

모든 대화를 계속 붙이면 토큰·비용·지연이 증가하고 오래된 잘못된 정보도 남습니다. 최근 메시지 window, 요약 메모리, 업무 DB의 장기 메모리를 구분합니다. 삭제 요청과 개인정보 보존 정책도 설계해야 합니다.

# 프롬프트 평가와 최적화

좋아 보이는 예시 하나가 아니라 고정 평가셋으로 version을 비교합니다.

평가 항목:

- task 정확도
- schema 준수율
- 금지 내용·hallucination 비율
- latency와 token 비용
- 긴 입력·모호한 입력·prompt injection 견고성

한 번에 여러 요소를 바꾸지 않고 prompt version, model, temperature를 기록합니다. LLM judge는 편향이 있으므로 중요한 표본은 사람이 검토합니다.

6. Prompt 개선은 무엇으로 비교하나요? **고정 평가셋과 version별 지표**
