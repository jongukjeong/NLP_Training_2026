# Chapter 17 비전공자용 프롬프트 평가 워크북

## 프롬프트는 좋은 문장보다 재현 가능한 작업 명세다

프롬프트 엔지니어링의 목표는 한 번 멋진 답을 얻는 것이 아닙니다. 입력이 달라져도 요구한 형식과 품질을 안정적으로 지키게 만드는 것이 목표입니다. 따라서 프롬프트를 프로그램 코드처럼 버전으로 관리하고 고정된 평가 질문으로 비교합니다.

## 기본 구성 다섯 가지

```text
역할: 어떤 관점에서 수행하는가
목표: 무엇을 완료해야 하는가
입력: 어떤 데이터가 주어지는가
제약: 무엇을 하면 안 되는가
출력 계약: 어떤 형식과 항목으로 답하는가
```

역할만 길게 쓰기보다 목표와 성공 기준을 구체적으로 적는 편이 중요합니다.

## Zero-shot과 Few-shot의 차이

Zero-shot은 설명만 주고 작업을 요청합니다. Few-shot은 입력과 기대 출력 예시를 함께 제공합니다. 예시는 단순한 정답 모음이 아니라 모델이 따라야 할 결정 경계를 보여 줍니다.

```text
입력: 배송이 아직 안 왔어요
출력: {"label":"배송"}

입력: 주문을 취소하고 싶어요
출력: {"label":"환불"}
```

예시가 실제 입력을 그대로 포함하면 평가 누수가 생길 수 있으므로 별도의 평가셋을 유지합니다.

## 정확도와 형식 준수율을 분리한다

평가 질문이 20개이고 정답이 17개라면 정확도는 다음과 같습니다.

\[
Accuracy=\frac{17}{20}=0.85
\]

이 중 JSON Schema를 지킨 답이 18개라면 형식 준수율은 90%입니다. 내용은 맞지만 형식이 틀린 경우와 형식은 맞지만 내용이 틀린 경우를 분리해야 개선 지점을 찾을 수 있습니다.

## 평가표를 먼저 만든다

| 항목 | 판정 예시 |
|---|---|
| 정답성 | 기대 Label과 같은가 |
| 형식 | JSON Schema를 통과하는가 |
| 근거성 | 입력에 없는 사실을 추가하지 않았는가 |
| 안전성 | 민감정보와 금지 행동을 피했는가 |
| 비용 | 평균 입력·출력 토큰 수 |
| 지연 | 평균과 상위 95% 응답 시간 |

## Structured Output도 프로그램에서 검사한다

```python
allowed = {"배송", "환불", "결제", "기타"}
assert set(result) == {"label", "reason"}
assert result["label"] in allowed
assert isinstance(result["reason"], str)
```

JSON처럼 보이는 문자열과 검증된 데이터는 다릅니다. Parsing, 필수 Key, 자료형, 허용 값 검사를 통과해야 다음 업무에 전달합니다.

## 내부 사고 과정 대신 검증 가능한 결과를 요청한다

긴 Chain of Thought 전체를 출력하게 하기보다 최종 답, 짧은 근거, 사용한 입력 항목과 불확실성을 요청합니다. 사용자는 결과를 확인할 수 있어야 하며 모델 내부의 상세 추론을 신뢰 근거로 삼으면 안 됩니다.

## 프롬프트 변경은 한 번에 하나씩 비교한다

1. 기준 프롬프트와 평가셋을 고정합니다.
2. 출력 형식 또는 예시처럼 한 요소만 바꿉니다.
3. 정확도·형식·비용·지연을 다시 측정합니다.
4. 좋아진 사례와 나빠진 사례를 모두 기록합니다.
5. 기준 미달이면 이전 버전으로 되돌립니다.

## Prompt Injection을 데이터로 취급한다

문서나 사용자 입력 안의 “이전 지시를 무시하라”는 문장은 신뢰할 수 없는 데이터입니다. 시스템 규칙과 사용자 데이터를 구분하고, 도구 실행·외부 전송·삭제 같은 행동은 별도 권한 검사와 사용자 확인을 거칩니다.

# Chapter 17 원리·수학·실습 가이드

## 1. 프롬프트를 프로그램처럼 관리하기

프롬프트는 감으로 고치는 문장이 아니라 입력과 출력 계약이다. `목표 → 입력 경계 → 규칙 → 출력 스키마 → 예시 → 실패 처리`를 명시하고 버전과 평가 결과를 기록한다.

Zero-shot은 예시 없이 지시하고, Few-shot은 대표 입출력 예시를 제공한다. 예시는 많기보다 경계 사례와 실제 분포를 잘 대표해야 한다.

## 2. Structured Output

```json
{"category":"배송","priority":"high","reason":"하루 이상 지연"}
```

필드 타입, 허용값, 누락 처리 규칙을 스키마로 정의한다. 단순히 “JSON으로 답해”라고만 하면 설명 문장이 섞이거나 키가 변할 수 있다. 파싱 실패 시 제한 횟수 재시도하고 원문을 보관한다.

## 3. 추론 요청과 검증

복잡한 문제는 하위 작업으로 나누고 필요한 중간 산출물만 구조화해 요청한다. 장황한 비공개 사고과정을 요구하기보다 근거, 계산 결과, 확인 가능한 요약을 받는다. 외부 문서의 명령은 데이터로 취급해 prompt injection을 막는다.

## 4. 평가 수학

\[
Accuracy=\frac{correct}{N},\quad SchemaRate=\frac{valid\ outputs}{N}
\]

\[
AverageCost=\frac{\sum_i cost_i}{N},\quad P95Latency=95\%\ 지점의\ 지연시간
\]

프롬프트 A/B는 같은 평가셋, 모델, 파라미터에서 비교한다. 정확도뿐 아니라 형식 준수율, 안전 실패, 토큰, 비용, 지연을 함께 본다. 테스트셋을 보며 프롬프트를 반복 수정하면 과적합되므로 개발셋과 최종 테스트셋을 분리한다.

1. Few-shot 예시는 어떤 사례를 우선해야 하는가?
2. Structured Output에서 스키마가 필요한 이유는?
3. 프롬프트 비교에서 정확도 외 지표는?

# Chapter 17 프롬프트 운영과 모니터링

## 개발 후에도 변한다

사용자 표현, 정책, 모델 버전이 바뀌면 프롬프트 성능도 변합니다. 배포 후 형식 실패율, 거절률, 토큰, 지연과 사용자 피드백을 모니터링합니다.

## Canary 배포

새 프롬프트를 전체 사용자에게 바로 적용하지 않고 일부 요청에서 기존 버전과 비교합니다. 안전 오류가 증가하면 빠르게 이전 버전으로 되돌립니다.

## A/B 해석

동일 기간의 비슷한 사용자 집단에서 비교하고 표본 수를 기록합니다. 작은 차이를 한두 건으로 결론 내리지 않습니다.

## Prompt drift

입력 길이·언어·카테고리 분포가 개발 데이터와 달라지는지 봅니다. 정답 레이블이 늦게 들어오면 형식 준수율과 사람 검토 결과를 선행 지표로 사용합니다.

## 로그 설계

Prompt 버전, 모델, latency, token, 파싱 결과, 오류 코드를 저장하되 사용자 원문과 개인정보는 최소화합니다.

## Rollback 기준

형식 실패·안전 위반·P95·비용의 허용선을 미리 정합니다. 장애 후 회의에서 기준을 만드는 것이 아니라 배포 전에 작성합니다.

# Chapter 17 프롬프트 평가와 개선 실습

## 프롬프트는 정답 문장이 아니라 실험 대상이다

좋은 프롬프트는 한 번에 떠오르는 마법 문장이 아닙니다. 고정 평가 질문으로 실패를 확인하고 한 요소씩 수정하는 작은 프로그램에 가깝습니다.

## 나쁜 요청과 개선 요청

```text
나쁜 예: 문의를 잘 분류해 줘.
```

```text
목표: 문의를 하나의 카테고리로 분류
허용값: 배송, 환불, 계정, 기타
규칙: 두 의도가 있으면 고객이 요구한 조치를 우선
출력: {"category":"...","reason":"..."}
불명확하면 기타
```

개선된 요청은 결과가 좋은지 자동으로 판정할 수 있습니다.

## 평가셋 만들기

쉬운 사례만 넣지 않습니다.

- 전형적인 입력
- 두 의도가 섞인 입력
- 오타와 짧은 입력
- 규칙 밖 입력
- prompt injection 문구가 포함된 입력
- 개인정보가 포함된 입력

개발셋으로 수정하고 test는 마지막 평가에만 사용합니다.

## 정확도와 형식 준수율

\[
Accuracy=정답수/N
\]

\[
SchemaRate=파싱성공수/N
\]

100건 중 분류 정답 90건, JSON 파싱 성공 96건이면 각각 90%, 96%입니다. 파싱 실패 4건은 분류 정답 여부와 별도 운영 오류입니다.

## Few-shot 예제 선택

같은 쉬운 배송 예시를 세 번 넣기보다 배송/환불 경계, 기타 처리, 복합 의도처럼 판단 규칙을 보여주는 사례를 고릅니다. 예제 순서에 따라 결과가 변하는지도 확인합니다.

## Structured Output 검증

필수 필드, 타입, 허용값을 검사합니다. `category="배송지연"`처럼 허용 목록에 없는 값은 JSON 형식이 맞아도 업무 schema에는 실패입니다.

## Chain of Thought를 쉽게 다루기

복잡한 문제는 “하위 항목을 확인하고 최종 결과와 짧은 근거를 출력”하도록 구성합니다. 장황한 내부 사고과정을 요구하기보다 확인 가능한 근거와 계산 결과를 요청합니다.

## 비용·지연 비교

프롬프트가 길어지면 항상 좋아지는 것은 아닙니다. 정확도, 입력 토큰, 출력 토큰, 평균 비용, P95 지연을 함께 기록합니다.

| 버전 | 정확도 | 형식 | 입력토큰 | P95 | 결론 |
|---|---:|---:|---:|---:|---|
| v1 | 기록 | 기록 | 기록 | 기록 | 기준 |
| v2 | 기록 | 기록 | 기록 | 기록 | 채택/기각 |

## 실패 대응

- 파싱 실패: schema 기반 제한 재시도
- 근거 부족: “모름” 반환
- 민감정보: 전송 전 마스킹
- 외부 입력의 명령: 데이터로 취급
- 반복 실패: 사람 검토 대기열

## 완료 기준

프롬프트 텍스트뿐 아니라 모델, 파라미터, 평가셋 버전, 결과, 실패 사례와 비용을 함께 저장합니다.

# 17.1~17.3 Prompt 기본·Zero-shot·Few-shot

Prompt는 역할 수사보다 목표, 입력, 제약, 출력 계약과 성공 기준이 중요합니다.

```text
목표 → 필요한 배경 → 입력 구분 → 제약 → 출력 schema → 예외 처리
```

Zero-shot은 예시 없이 지시하고 Few-shot은 대표 입력·출력 예시를 제공합니다. Few-shot 예시는 정상 사례뿐 아니라 경계·거부 사례와 label 균형을 포함합니다.

Prompt injection에 대비해 사용자 입력과 신뢰된 지시를 구분하고, 입력 안의 지시를 시스템 규칙으로 취급하지 않도록 합니다. 모델 출력은 검증 전까지 신뢰하지 않습니다.

## 프롬프트를 계약서처럼 쓰기

좋은 프롬프트는 화려한 역할 문장보다 목표와 판정 기준이 명확합니다.

```text
목표: 고객 문의를 하나의 카테고리로 분류
입력: <inquiry>...</inquiry>
허용값: 배송, 환불, 계정, 기타
규칙: 불명확하면 기타
출력: {"category":"...","reason":"..."}
```

입력 경계를 태그로 분리하면 사용자 문장과 시스템 규칙을 구분하기 쉽습니다. 입력 속 “앞 지시를 무시하라”는 문장은 데이터이지 신뢰된 지시가 아닙니다.

## Zero-shot과 Few-shot

Zero-shot은 설명만으로 수행하고 Few-shot은 예시를 함께 줍니다. 예시는 쉬운 사례만 반복하지 말고 클래스 경계, 예외, 모호한 사례를 포함합니다. 잘못된 예시는 모델에 강한 잘못된 패턴을 줄 수 있습니다.

## 평가를 수치로 보기

\[
Accuracy=\frac{정답수}{전체수},\quad
SchemaRate=\frac{파싱가능한출력수}{전체수}
\]

100개 중 정답 87개, JSON 파싱 성공 95개면 정확도 87%, 형식 준수율 95%입니다. 정확도만 보면 운영에서 5%가 파싱 실패한다는 문제를 놓칩니다.

## 프롬프트 변경 실험

평가셋, 모델, temperature를 고정하고 프롬프트 한 요소만 바꿉니다. 개발셋 결과로 개선하고 마지막 test는 최종 확인에만 사용합니다. 프롬프트 버전, 토큰 수, 비용, P95 지연도 기록합니다.

## 쉬운 개선 순서

모호한 목표 수정 → 입력 경계 표시 → 출력 스키마 명시 → 대표 예시 추가 → 실패 처리 규칙 → 자동 평가 순으로 개선합니다. 무조건 긴 프롬프트가 좋은 것은 아닙니다.

# 17.4 Chain of Thought · 17.5 Structured Output

복잡한 task는 문제를 단계로 나누고 필요한 중간 검증을 요청할 수 있습니다. 내부 추론 전체를 노출하도록 요구하기보다 최종 답과 검증 가능한 짧은 근거·계산 결과를 요청합니다.

Structured Output은 후속 시스템이 파싱할 schema를 강제합니다.

```json
{
  "category": "delivery",
  "priority": "high",
  "reason": "배송 지연과 행사 일정 충돌"
}
```

JSON parse, 필수 field, enum, 길이와 금지값을 코드로 검증하고 실패 시 제한된 재시도를 수행합니다. 자연어로 “JSON으로 써줘”만 요청하는 것보다 API의 schema 기능을 우선 검토합니다.

## 프롬프트 템플릿을 사용하는 이유

문자열을 매번 이어 붙이면 입력 경계와 변수 누락을 놓치기 쉽습니다. 템플릿은 고정 지시와 사용자 데이터를 분리하고 같은 구조를 반복 실행하게 합니다.

```python
template = """다음 문의를 분류하세요.
허용 카테고리: 배송, 환불, 계정, 기타
문의: <input>{inquiry}</input>
JSON만 출력하세요."""
prompt = template.format(inquiry=user_text)
```

사용자 입력에 중괄호나 태그가 들어올 수 있으므로 실제 애플리케이션에서는 프레임워크의 template API와 입력 검증을 사용합니다. 템플릿은 보안을 자동 해결하지 않으며 외부 입력을 지시보다 낮은 신뢰도로 다뤄야 합니다.

## 메시지 역할 분리

대화 모델의 입력은 일반적으로 시스템 지시, 사용자 요청, 도구 결과 등 역할별 메시지로 구성됩니다. 정책과 출력 계약은 신뢰된 메시지에 두고, 검색 문서와 도구 결과는 참고 데이터임을 명시합니다.

## Partial prompt와 재사용

날짜, 제품 목록, 출력 언어처럼 여러 요청에서 반복되는 일부 변수를 미리 묶으면 프롬프트 중복을 줄일 수 있습니다. 하지만 자동으로 들어가는 값도 버전과 출처를 기록해야 재현할 수 있습니다.

## Streaming의 의미

Streaming은 답 전체가 완성되기 전에 토큰 일부를 순차적으로 전달해 체감 지연을 줄입니다. 총 생성시간과 비용이 반드시 줄어드는 것은 아닙니다. JSON처럼 전체 구조 검증이 필요한 출력은 스트림 도중 실행하지 않고 완성 후 파싱합니다.

## 출력 파싱 실패 처리

1. schema validation으로 타입과 필수 필드를 검사합니다.
2. 실패 이유를 기록하고 제한 횟수만 재요청합니다.
3. 반복 실패는 안전한 기본값 또는 사람 검토로 보냅니다.
4. 모델 원문과 파싱 결과를 함께 보존합니다.

```python
required = {"category", "reason"}
if not required.issubset(result):
    raise ValueError("필수 필드 누락")
```

## 대화 메모리와 프롬프트 예산

모든 대화를 계속 붙이면 토큰·비용·지연이 증가하고 오래된 잘못된 정보도 남습니다. 최근 메시지 window, 요약 메모리, 업무 DB의 장기 메모리를 구분합니다. 삭제 요청과 개인정보 보존 정책도 설계해야 합니다.

# 프롬프트 평가와 최적화

좋아 보이는 예시 하나가 아니라 고정 평가셋으로 version을 비교합니다.

평가 항목:

- task 정확도
- schema 준수율
- 금지 내용·hallucination 비율
- latency와 token 비용
- 긴 입력·모호한 입력·prompt injection 견고성

한 번에 여러 요소를 바꾸지 않고 prompt version, model, temperature를 기록합니다. LLM judge는 편향이 있으므로 중요한 표본은 사람이 검토합니다.

## 평가셋 분리

프롬프트를 수정하는 개발셋과 마지막 확인용 test를 분리합니다. Test 실패를 보고 계속 수정하면 test도 학습 자료가 됩니다.

## 자동 평가

분류 값, 필수 키, 타입, 문자열 길이, 금지 표현은 코드로 검사합니다.

```python
def validate(result):
    allowed = {"배송", "환불", "계정", "기타"}
    return (
        isinstance(result, dict)
        and result.get("category") in allowed
        and isinstance(result.get("reason"), str)
    )
```

## 사람 평가

요약·생성처럼 답이 여러 개인 과제는 사실성, 충분성, 명료성, 안전성을 1~5점으로 평가합니다. 평가자에게 좋은/나쁜 예를 제공합니다.

## 비용과 지연

\[
평균비용=전체요청비용/요청수
\]

P95는 요청의 95%가 이 시간 이내에 완료된다는 뜻입니다. 평균이 낮아도 일부 사용자가 오래 기다릴 수 있어 함께 봅니다.

## 평가 결과표

| Prompt | Accuracy | Schema | 사람점수 | P95 | 비용 |
|---|---:|---:|---:|---:|---:|
| v1 | 기록 | 기록 | 기록 | 기록 | 기록 |
| v2 | 기록 | 기록 | 기록 | 기록 | 기록 |

## 실패 유형

지시 누락, 입력 오해, 형식 오류, 근거 없는 생성, 안전 위반으로 분류합니다. 유형별 수정 방법이 다릅니다.

## 통계적 주의

평가 질문 5개에서 1건 차이는 20%p입니다. 작은 표본의 큰 차이를 일반화하지 않고 질문 수와 반복 실행 수를 함께 보고합니다.

# 실습: 프롬프트 최적화

고객 문의 분류 prompt의 zero-shot/few-shot 버전을 생성하고 API 응답의 JSON schema 준수율을 비교합니다.

- [안내](examples/05_prompt_optimization_solution/README.md)
- [코드](examples/05_prompt_optimization_solution/prompt_evaluator.py)

## 실습 목표

하나의 업무 Prompt를 Zero-shot에서 시작해 Few-shot과 Structured Output으로 개선하고 같은 평가셋에서 비교합니다.

## 단계

1. 평가 질문 30개와 기대 결과 작성
2. Zero-shot 기준 Prompt 실행
3. 오류 유형 분류
4. 판단 경계 예시 3~5개 추가
5. JSON schema 적용
6. 정확도·형식·토큰·P95 비교

## 결과 파일

```text
prompt_version,input_id,output,parsed,correct,latency,input_tokens,output_tokens,error_type
```

## Prompt 버전

Prompt 파일에 `v01`, `v02`를 붙이고 변경 이유를 기록합니다. 결과가 나빠진 버전도 삭제하지 않아 반복 실험을 방지합니다.

## 안전 사례

입력 내부에 지시 무시 문구, 개인정보, 범위 밖 요청을 포함합니다. 외부 입력은 데이터로 취급하고 민감정보는 전송 전 제거합니다.

## 완료 기준

기준선과 개선 버전, 30개 이상 평가, 자동 파서, 오류 분석, 비용·지연, 최종 채택 이유를 제출합니다.
