# Chapter 19 비전공자용 AI Agent 신뢰성 워크북

## Agent는 답변에서 행동으로 범위를 넓힌다

일반 챗봇은 텍스트를 반환하지만 Agent는 도구를 선택하고 인자를 만들며 결과를 읽어 다음 단계를 결정할 수 있습니다. 따라서 답변 오류뿐 아니라 잘못된 조회, 중복 실행, 데이터 변경과 외부 전송 위험을 관리해야 합니다.

```text
목표 확인 → 도구 선택 → 인자 검증 → 실행 → 결과 검증 → 종료 판단
```

## Function Calling은 실행이 아니라 제안이다

모델이 도구 이름과 인자를 만들었다고 해서 자동으로 실행하면 안 됩니다. 애플리케이션이 Schema, 권한, 허용 범위와 사용자 확인을 검사한 뒤 실행 여부를 결정합니다.

```json
{
  "name": "lookup_order",
  "arguments": {"order_id": "A-1024"}
}
```

`order_id` 형식과 현재 사용자가 그 주문을 조회할 권한이 있는지 프로그램에서 검사합니다.

## 읽기와 쓰기 도구를 분리한다

- 읽기: 검색, 조회, 상태 확인
- 쓰기: 전송, 수정, 삭제, 결제, 예약

쓰기 작업은 미리보기, 명시적 사용자 승인, 실행 결과와 감사 로그가 필요합니다. 삭제처럼 되돌리기 어려운 행동은 더 강한 확인 절차를 적용합니다.

## 단계가 늘수록 전체 성공률은 낮아질 수 있다

각 단계 성공률이 95%이고 서로 독립이라고 단순 가정하면 5단계 모두 성공할 확률은 다음과 같습니다.

\[
0.95^5\approx0.774
\]

10단계라면 약 0.599입니다. 실제 시스템은 더 복잡하지만, 불필요한 단계를 줄이고 단계별 검증과 재시도를 설계해야 한다는 점을 보여 줍니다.

## 무한 반복을 막는 종료 조건

- 최대 단계 수
- 최대 토큰과 비용
- 같은 도구·인자의 반복 감지
- 연속 실패 횟수
- 목표 달성 또는 사용자 확인 필요 상태

종료 이유를 로그에 남겨야 실패를 재현할 수 있습니다.

## 재시도와 중복 실행을 구분한다

네트워크 오류 뒤 같은 결제 요청을 다시 보내면 두 번 처리될 수 있습니다. 쓰기 도구에는 Idempotency Key를 사용하고, 재시도 전에 이전 요청의 처리 상태를 확인합니다.

## 상태 그래프는 가능한 흐름을 명시한다

LangGraph 같은 상태 기반 접근에서는 현재 상태, 실행할 Node와 다음 Edge를 정의합니다. 성공·재시도·사용자 승인·실패 종료 경로를 그림으로 확인할 수 있어 복잡한 Workflow에 유리합니다.

## MCP는 연결 규약이다

MCP는 모델 자체가 아니라 외부 기능과 정보를 일관된 방식으로 노출하기 위한 Protocol입니다.

- Resources: 읽을 수 있는 정보
- Tools: 실행 가능한 기능
- Prompts: 재사용 가능한 상호작용 템플릿

MCP를 사용해도 인증, 권한, 입력 검증과 감사는 애플리케이션의 책임입니다.

## Agent 평가 사례

| 평가 항목 | 측정 예시 |
|---|---|
| 목표 성공률 | 과업을 올바르게 완료한 비율 |
| 도구 선택 정확도 | 필요한 도구를 선택한 비율 |
| 인자 정확도 | Schema와 업무 규칙을 지킨 비율 |
| 불필요한 호출 | 목표와 무관한 호출 수 |
| 안전 위반 | 승인 없는 쓰기나 정보 노출 수 |
| 비용·지연 | 요청당 토큰, 호출 수, 완료 시간 |

## 실패 후 확인할 순서

목표 해석, 도구 선택, 인자 생성, 권한 검사, 도구 실행, 결과 해석 중 어느 단계에서 처음 잘못됐는지 찾습니다. 마지막 답만 보면 근본 원인을 놓치기 쉽습니다.
