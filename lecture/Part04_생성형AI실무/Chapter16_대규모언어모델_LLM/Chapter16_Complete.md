# Chapter 16 LLM 벤치마크와 용량 계획

## 업무 벤치마크

공개 순위보다 우리 질문을 잘 푸는지가 중요합니다. 정답형, 요약, 구조화 출력, 긴 문맥, 안전 거절을 포함한 50개 이상의 평가셋을 만듭니다.

## 동시 사용자

한 요청 속도만으로 운영 용량을 판단하지 않습니다. 동시 1·2·4·8 요청에서 TTFT와 처리량을 측정합니다. 동시성이 늘면 각 사용자 지연이 증가할 수 있습니다.

## 처리량

\[
Throughput=완료요청수/측정시간
\]

10분 동안 300건을 처리하면 평균 0.5 requests/s입니다. 평균뿐 아니라 P95 지연과 오류율을 함께 봅니다.

## 모델 라우팅

단순 분류는 작은 모델, 복잡한 생성은 큰 모델로 보낼 수 있습니다. Router 오류와 추가 지연을 포함해 전체 품질·비용을 평가합니다.

## 용량 여유

평균 사용량에 딱 맞추지 않고 피크 요청, 긴 입력, 모델 재시작을 고려합니다. 메모리 부족으로 프로세스가 종료되는 경우 복구와 대기열 정책을 둡니다.

## 결과표

| 모델 | 정답률 | TTFT P95 | tok/s | 동시4 오류 | 메모리 |
|---|---:|---:|---:|---:|---:|
| 후보 A | 기록 | 기록 | 기록 | 기록 | 기록 |
| 후보 B | 기록 | 기록 | 기록 | 기록 | 기록 |

# Chapter 16 로컬 LLM 선택과 운영 실습

## 모델 선택을 자동차 선택으로 이해하기

가장 큰 모델이 항상 최선은 아닙니다. 대형 승용차가 넓지만 연료와 주차 공간이 더 필요한 것처럼, 큰 모델은 메모리·전력·지연 비용이 큽니다. 실제 질문을 얼마나 잘 해결하는지와 운영 자원을 함께 비교합니다.

## 실행 전 확인표

```text
업무: 요약 / 질의응답 / 코드 / 분류
언어: 한국어 비중
장비: CPU, RAM, GPU, VRAM
요청량: 동시 사용자와 하루 요청 수
보안: 외부 API 전송 가능 여부
목표: 정확도, 첫 토큰 지연, 비용
```

## 메모리 손계산

\[
가중치메모리\approx 파라미터수\times 비트수/8
\]

7B 모델을 FP16으로 저장하면 `70억×16/8≈14GB`, 4-bit라면 이론상 약 3.5GB입니다. 실제 실행은 KV cache와 버퍼가 더 필요합니다.

## Context가 메모리를 쓰는 이유

모델은 생성 중 과거 토큰의 Key와 Value를 KV cache에 보관합니다. 긴 문서와 여러 사용자를 동시에 처리하면 이 영역이 커집니다. 모델 파일만 VRAM에 들어간다고 운영 가능하다고 결론 내리지 않습니다.

## 첫 실행과 반복 실행

첫 요청은 모델 로딩 때문에 느립니다. 이를 cold start라 합니다. 이미 메모리에 올라온 뒤의 warm run과 분리해 측정합니다.

| 측정 | 의미 |
|---|---|
| Load time | 모델 준비 시간 |
| TTFT | 첫 토큰까지 시간 |
| tokens/s | 생성 속도 |
| Peak RAM/VRAM | 최고 메모리 |
| 성공률 | 평가 질문 정답 비율 |

## 양자화 비교

동일 질문 30개를 FP16, 8-bit, 4-bit에서 실행합니다. 단순히 “답이 자연스럽다”가 아니라 정답성·형식·지연·메모리를 표로 기록합니다.

```text
모델/양자화 | Hit | 형식준수 | TTFT | tok/s | VRAM
```

## 로컬 서버 보안

기본 포트가 외부 네트워크에 열려 있지 않은지 확인합니다. 인증 없는 모델 서버가 공개되면 누구나 자원을 사용하거나 민감한 입력을 보낼 수 있습니다. 방화벽, 인증, 접근 로그와 입력 보존 정책을 설정합니다.

## 모델 파일과 라이선스

모델마다 상업적 사용, 재배포, 사용자 수 제한이 다를 수 있습니다. 모델 카드와 라이선스를 프로젝트 문서에 기록합니다. 모델 가중치는 용량이 크므로 Git 저장소에 커밋하지 않습니다.

## 품질이 낮을 때 점검 순서

1. 모델의 한국어·업무 적합성
2. Prompt와 출력 형식
3. context에 필요한 근거가 포함됐는지
4. quantization에 따른 품질 변화
5. 최대 출력 길이와 중단 조건
6. 모델 변경 전 평가셋 일관성

## 쉬운 결과 설명

“4-bit 모델이 좋았다” 대신 “메모리가 9GB에서 5GB로 줄고 속도는 18→24 tokens/s로 증가했지만, 30개 질문 정답은 26개에서 23개로 감소했다”고 설명합니다.

# Chapter 16 원리·수학·실습 가이드

## 1. LLM을 비교하는 기준

LLM은 대규모 텍스트에서 다음 토큰 예측을 학습한 언어모델이다. GPT, Llama, Gemma, Mistral, Qwen은 라이선스, 지원 언어, 문맥 길이, 도구 사용, 배포 방식이 다르다. 이름이나 파라미터 수만으로 선택하지 말고 실제 업무 평가셋으로 품질·지연·비용을 비교한다.

## 2. 메모리의 기초 계산

가중치만 저장하는 최소 메모리는 대략 다음과 같다.

\[
Memory\approx Parameters\times Bits/8
\]

70억 파라미터 모델은 FP16이면 약 `7B×2 byte=14GB`, 4-bit이면 이론상 약 3.5GB다. 실제 실행에는 KV cache, 활성값, 런타임 버퍼가 추가된다.

## 3. Quantization과 문맥 비용

양자화는 가중치를 적은 비트로 표현해 메모리와 전송량을 줄인다. 속도가 항상 같은 비율로 빨라지지는 않으며 작은 값 구분이 줄어 품질이 하락할 수 있다. 긴 문맥은 KV cache를 늘리고 일반 Attention 계산은 길이에 대해 대략 제곱으로 증가한다.

## 4. 로컬 실행 실험

```text
고정할 것: 모델/리비전, 양자화, 프롬프트, seed, 최대 토큰
측정할 것: 첫 토큰 지연, tokens/s, peak memory, 정답률, 형식 준수율
```

동일 질문 20개 이상으로 모델별 결과를 저장한다. 라이선스, 모델 카드, 데이터 외부 전송 금지 요구, GPU 메모리를 선택표에 포함한다. 모델 파일은 저장소에 커밋하지 않는다.

1. 8B 모델 FP16 가중치의 이론상 최소 메모리는?
2. 4-bit 양자화의 장점과 위험은?
3. 모델 선택에서 파라미터 수 외에 무엇을 측정할까?

# 16.1~16.6 LLM과 주요 계열

LLM은 대규모 말뭉치와 파라미터로 다음 token 예측 등을 학습한 언어모델입니다. 모델 이름만으로 품질을 판단하지 않고 task, 언어, context, tool use, 라이선스, hardware와 평가 결과를 비교합니다.

| 계열 | 검토 관점 |
|---|---|
| GPT | API 기반 frontier·도구 사용과 관리형 서비스 |
| Llama | 공개 weight 생태계와 다양한 fine-tune |
| Gemma | Google 계열 공개 모델과 크기 선택 |
| Mistral | 효율적인 공개·상용 모델 선택지 |
| Qwen | 다국어·코딩 등 폭넓은 크기와 task 계열 |

“오픈 모델”은 라이선스가 모두 같다는 뜻이 아닙니다. 상업 이용, 재배포, 파생 모델과 acceptable-use 조건을 모델별로 확인합니다.

평가 축: task 품질, 한국어, latency, throughput, memory, context 길이, hallucination, 안전성, 총비용.

## 큰 모델을 숫자로 비교하기

파라미터는 모델이 학습한 숫자의 개수입니다. 가중치만 저장하는 이론상 메모리는 다음처럼 추정합니다.

\[
Memory\approx Parameters\times\frac{bits}{8}
\]

8B 모델은 FP16에서 약 `80억×2 byte=16GB`, 4-bit에서 약 4GB입니다. 실제 실행에는 KV cache와 런타임 버퍼가 더 필요하므로 GPU 메모리와 정확히 같지 않습니다.

## 모델 계열을 보는 공통 기준

GPT, Llama, Gemma, Mistral, Qwen을 이름으로 서열화하지 않습니다. 지원 언어, 라이선스, 문맥 길이, 도구 호출, 양자화 지원, 하드웨어, 실제 평가 정확도와 지연을 표로 비교합니다. 같은 계열에서도 버전과 크기에 따라 조건이 달라집니다.

## 토큰과 문맥 길이

문맥 길이는 글자 수가 아니라 토큰 수입니다. 한국어는 tokenizer에 따라 같은 문장이 다른 토큰 수가 될 수 있습니다. 입력, 시스템 지시, 검색 문맥, 출력 토큰이 모두 문맥 예산을 사용합니다.

## KV cache의 직관

생성할 때 과거 모든 토큰의 Key와 Value를 매번 다시 계산하지 않도록 저장한 것이 KV cache입니다. 문맥과 batch가 길어질수록 메모리가 증가합니다. 가중치가 들어간다고 실행 가능한 것이 아닌 이유입니다.

## Quantization 해석

4-bit 양자화는 연속적인 가중치를 제한된 단계로 근사합니다. 메모리는 줄지만 작은 값 차이가 사라져 품질이 낮아질 수 있습니다. 속도 향상은 하드웨어와 커널 지원에 따라 달라 실제 측정이 필요합니다.

## 선택 실험

최소 20~50개의 업무 질문으로 정답성, 형식 준수율, 첫 토큰 지연, tokens/s, peak memory를 동일 조건에서 측정합니다. 모델이 크다는 이유만으로 업무 품질이 높다고 결론 내리지 않습니다.

# 로컬 LLM 실행과 운영 판단

로컬 실행 장점은 데이터 경계와 실험 통제이며, 단점은 hardware·배포·보안 패치·모델 관리 책임입니다.

Ollama 기본 흐름:

```text
모델 pull → local API 실행 → prompt 요청 → token/latency 기록
```

정량 기록:

- prompt/output token 수
- load/prompt evaluation/generation 시간
- tokens per second
- peak memory
- model tag와 digest
- generation parameters

양자화는 memory를 줄일 수 있지만 품질과 속도 변화가 있으므로 동일 평가셋으로 비교합니다. 민감 데이터가 로컬이라는 이유만으로 안전한 것은 아니며 로그, cache, model access와 OS 권한을 관리해야 합니다.

## 로컬 LLM은 무엇이 다른가?

클라우드 API는 외부 서버에 요청하고 사용량만큼 비용을 내는 방식입니다. 로컬 LLM은 모델 파일을 직접 내려받아 자신의 CPU/GPU에서 실행합니다. 데이터 통제와 오프라인 실행 장점이 있지만 설치·메모리·업데이트·보안 책임도 직접 가집니다.

## 실행 전 자원 계산

모델 가중치 예상 메모리에 여유 공간과 KV cache를 더합니다. 8B 4-bit가 이론상 약 4GB라도 실제로는 4GB GPU에 안정적으로 들어간다고 보장할 수 없습니다.

```text
필요 메모리 ≈ 가중치 + KV cache + runtime buffer + 여유 공간
```

CPU offload는 GPU에 못 들어간 일부를 시스템 메모리에서 처리하지만 속도가 크게 느려질 수 있습니다.

## Ollama 방식의 쉬운 흐름

```text
런타임 설치 → 모델 선택/다운로드 → 로컬 서버 실행
→ CLI 또는 HTTP 요청 → 속도·메모리·품질 측정
```

모델 태그에는 크기와 양자화가 포함될 수 있으므로 이름만 같다고 동일 파일로 보지 않습니다. 교육 자료에서는 특정 모델 다운로드 명령보다 `ollama list`, 모델 정보와 실제 버전을 기록하는 습관을 강조합니다.

## 성능 지표

- TTFT: 요청 후 첫 토큰이 나오기까지 시간
- tokens/s: 초당 생성 토큰
- Peak memory: 최고 메모리 사용량
- Context length: 실제 입력 가능한 토큰 범위
- 품질: 고정 평가 질문의 정답성과 형식 준수

사용자는 첫 토큰 지연에 민감하고, 긴 보고서 생성은 tokens/s가 중요할 수 있습니다.

## 동일 조건 비교

프롬프트, 최대 출력 토큰, temperature, 입력 길이, 하드웨어를 고정합니다. 첫 실행은 모델 로딩 때문에 느릴 수 있어 cold start와 warm run을 나눠 측정합니다.

## 개인정보와 로그

로컬이라고 자동으로 안전한 것은 아닙니다. 프롬프트 로그, 모델 서버 접근 포트, 다운로드 모델의 출처, 라이선스, 악성 파일 위험을 관리합니다. 서버를 외부 네트워크에 노출할 때 인증과 방화벽을 적용합니다.

## 실패 문제 해결

| 증상 | 확인 항목 |
|---|---|
| 모델 로딩 실패 | RAM/VRAM, 파일 손상, runtime 버전 |
| 매우 느림 | CPU fallback, context 길이, 양자화/커널 |
| 답변 반복 | generation 설정, prompt, 모델 품질 |
| 한글 품질 낮음 | tokenizer와 학습 언어, 모델 선택 |
| 메모리 급증 | batch, context, KV cache |

## 운영 선택 질문

# 실습: 로컬 LLM 실행

- [안내](examples/04_local_llm_solution/README.md)
- [코드](examples/04_local_llm_solution/local_llm_runner.py)
- [입력](examples/04_local_llm_solution/prompts.csv)

Ollama 모델명을 환경변수로 받아 응답과 성능 metadata를 JSONL로 기록합니다.

## 실습 준비

로컬 런타임과 모델을 선택하기 전에 장비 정보를 기록합니다.

```text
OS:
CPU/RAM:
GPU/VRAM:
Runtime 버전:
모델 태그·양자화:
```

모델 파일 크기와 실제 peak memory는 다르므로 실행 중 값을 측정합니다.

## 고정 Prompt 세트

요약, 분류, 한국어 질의응답, 구조화 출력, 긴 문맥 질문을 최소 20개 준비합니다. 모델마다 같은 입력과 최대 출력 길이, temperature를 사용합니다.

## 측정 코드의 핵심

```python
started = time.perf_counter()
result = generate(prompt)
elapsed = time.perf_counter() - started
```

가능하면 첫 토큰 시간과 전체 시간을 분리합니다. 첫 실행은 모델 로딩을 포함하므로 cold와 warm 실행을 나눕니다.

## 결과 CSV

```text
model,quantization,prompt_id,passed,ttft,total_seconds,tokens_per_second,error
```

정답성은 필수 키워드, JSON schema, 사람 평가 기준으로 판정합니다.

## 동시성 실험

동시 요청 1·2·4개에서 P50/P95와 오류율을 비교합니다. 한 요청이 빠른 것과 여러 사용자를 안정적으로 처리하는 것은 다릅니다.

## 안전 점검

모델 서버가 외부 인터페이스에 인증 없이 열리지 않았는지 확인하고 Prompt 로그의 개인정보를 마스킹합니다. 모델 라이선스와 출처도 기록합니다.

## 완료 기준

두 개 이상 모델 또는 양자화 설정, 20개 이상 질문, 품질·TTFT·tokens/s·메모리 표, 실패 사례와 선택 결론을 제출합니다.
