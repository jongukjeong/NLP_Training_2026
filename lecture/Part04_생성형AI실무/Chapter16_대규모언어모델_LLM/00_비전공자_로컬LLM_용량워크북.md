# Chapter 16 비전공자용 로컬 LLM 용량 워크북

## 모델 선택은 가장 큰 모델을 고르는 문제가 아니다

좋은 모델은 현재 업무의 품질 기준을 만족하면서 보유 장비에서 필요한 속도와 비용으로 실행되는 모델입니다. 모델 이름보다 언어, 과제, 라이선스, 메모리, 지연 시간과 평가 결과를 함께 봅니다.

## 파라미터 메모리를 먼저 손으로 계산하기

파라미터가 (N)개이고 파라미터당 (b) bit를 사용하면 가중치 메모리의 단순 근사는 다음과 같습니다.

\[
Memory\approx\frac{N\times b}{8}
\]

7B 모델을 FP16으로 저장하면 다음과 같습니다.

\[
7\times10^9\times16/8\approx14GB
\]

4bit 양자화의 이론적 값은 약 3.5GB입니다. 실제 실행에는 메타데이터, 임시 Buffer와 KV Cache가 필요하므로 더 많은 여유 공간을 잡아야 합니다.

## 양자화는 숫자를 더 적은 bit로 저장한다

- FP16: 품질 보존에 유리하지만 메모리 사용량이 큼
- 8bit: 메모리를 줄이면서 비교적 안정적
- 4bit: 로컬 실행에 유리하지만 모델·과제별 품질 확인 필요

양자화는 압축 파일처럼 실행 전에 원상 복원하는 방식이 아닙니다. 더 낮은 정밀도의 숫자로 계산하므로 속도와 품질이 달라질 수 있습니다.

## Context가 길어지면 KV Cache도 커진다

생성 중에는 이전 토큰의 Key와 Value를 저장해 같은 계산을 반복하지 않습니다. 이 저장 공간이 KV Cache입니다. Batch, 토큰 길이, 층 수와 차원이 커질수록 메모리 사용량이 증가합니다.

단순히 모델 파일이 VRAM에 들어간다고 실행이 보장되지 않습니다. 긴 문맥과 동시 사용자를 고려해야 합니다.

## TTFT와 생성 속도를 구분하기

- TTFT: 요청 후 첫 토큰이 나올 때까지의 시간
- Tokens/s: 이후 1초에 생성하는 토큰 수
- End-to-end latency: 전체 답변이 끝날 때까지의 시간
- Throughput: 일정 시간 동안 처리한 전체 요청 또는 토큰 수

사용자는 첫 반응과 전체 완료를 다르게 느끼므로 두 지표를 함께 기록합니다.

## 같은 조건으로 모델을 비교하기

1. 동일한 장비와 실행 도구를 사용합니다.
2. Prompt와 생성 설정을 고정합니다.
3. 첫 실행과 반복 실행을 구분합니다.
4. 품질, TTFT, Tokens/s, Peak memory를 기록합니다.
5. 성공 사례뿐 아니라 실패 질문도 보존합니다.

## 모델 계열 이름만으로 품질을 단정하지 않는다

Llama, Gemma, Mistral, Qwen 등은 서로 다른 크기와 목적의 모델군입니다. 같은 계열에서도 Base와 Instruction 모델, 버전, 언어 성능, Context 길이와 라이선스가 다릅니다. 현재 시점의 모델 카드와 공식 라이선스를 확인해야 합니다.

## 로컬 실행의 장점과 책임

장점은 데이터 경계를 직접 통제하고 네트워크 연결 없이 실험할 수 있다는 것입니다. 반대로 모델 파일 관리, 보안 패치, 접근 제어, 로그, 용량 계획과 장애 대응 책임도 직접 맡습니다.

## 결과 설명 예시

```text
4bit 모델은 현재 GPU에서 실행됐으며 평균 생성 속도는 실험 조건에서 측정했다.
다만 긴 문서 요약에서 누락이 증가했으므로 품질 기준을 만족한다고 단정하지 않는다.
다음 실험에서는 Context 길이와 양자화 수준을 분리해 비교한다.
```
