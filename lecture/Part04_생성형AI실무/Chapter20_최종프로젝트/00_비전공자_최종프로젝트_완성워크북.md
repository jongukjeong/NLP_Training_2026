# Chapter 20 비전공자용 최종 프로젝트 완성 워크북

## 프로젝트는 기술 이름이 아니라 해결할 문제로 시작한다

“LLM과 RAG를 사용한다”는 구현 방법일 뿐 프로젝트 목표가 아닙니다. 누가 어떤 상황에서 무엇을 더 빠르고 정확하게 해결하는지를 한 문장으로 정의합니다.

```text
고객지원 담당자가 최신 정책 문서를 근거로
문의 답변 초안을 빠르게 작성하도록 돕는다.
```

## 범위를 작게 정하면 평가가 가능해진다

- 포함: 정책 검색, 출처 표시, 답변 초안
- 제외: 자동 환불 승인, 고객에게 자동 전송
- 사용자: 내부 고객지원 담당자
- 데이터: 공개 또는 비식별 정책 문서
- 실패 시 행동: 근거 부족 표시와 담당자 확인 요청

## 성공 기준을 시스템 단계별로 나눈다

| 단계 | 지표 예시 |
|---|---|
| 검색 | Recall@3, MRR |
| 생성 | 근거성, 완전성, 인용 정확도 |
| 서비스 | 오류율, 응답 시간, 가용성 |
| 사용자 | 수정 시간, 채택률, 만족도 |
| 안전 | 권한 위반, 민감정보 노출 건수 |

전체 만족도만 측정하면 어디를 개선해야 할지 알기 어렵습니다.

## 기준선부터 만든다

복잡한 모델 전에 키워드 검색, 고정 답변 또는 단순 분류기로 기준선을 만듭니다. 새 시스템이 기준선보다 얼마나 개선됐는지 보여 줄 수 있어야 기술 선택의 이유가 생깁니다.

## 데이터 분할은 실제 사용 시점을 흉내 낸다

무작위 분할만 사용하면 거의 같은 문서나 질문이 학습과 평가에 동시에 포함될 수 있습니다. 문서 ID, 사용자, 시간과 템플릿 중 실제 서비스에서 새로 나타나는 단위를 기준으로 분리합니다.

## 비용을 요청 단위로 계산한다

요청 한 건의 입력 토큰이 2,000개, 출력이 500개이고 단가가 각각 (C_{in}, C_{out})이라면 다음과 같습니다.

\[
Cost_{request}=2000C_{in}+500C_{out}
\]

월간 요청이 (R)건이면 단순 추정 월 비용은 (R\times Cost_{request})입니다. 검색·재시도·Embedding·저장소·모니터링 비용도 별도로 더합니다. 실제 단가는 사용하는 서비스의 현재 가격표로 계산합니다.

## 지연 시간도 단계별로 측정한다

\[
T_{total}=T_{load}+T_{retrieve}+T_{rerank}+T_{generate}+T_{postprocess}
\]

평균만 보지 말고 중앙값과 상위 95% 값을 함께 기록합니다. 일부 느린 요청이 사용자 경험을 크게 해칠 수 있습니다.

## 실험 기록은 한 줄로 재현 가능해야 한다

| 항목 | 기록 내용 |
|---|---|
| 실험 ID | 날짜와 순번 |
| 변경점 | Chunk 크기, 모델, Prompt 등 한 가지 |
| 고정 조건 | 평가셋, Seed, 검색 k |
| 결과 | 품질·비용·지연 지표 |
| 실패 사례 | 대표 입력과 원인 |
| 결정 | 채택, 보류, 폐기와 이유 |

## 최소 서비스 계약을 정의한다

```json
{
  "question": "환불 가능 기간은?",
  "answer": "상품 수령 후 7일 이내입니다.",
  "sources": [{"document_id": "policy-12", "page": 3}],
  "needs_review": false
}
```

입력 제한, Timeout, 오류 응답과 출처 형식을 API 계약에 포함합니다.

## 데모 실패도 발표의 일부로 준비한다

- 고정 질문과 예상 결과를 준비합니다.
- 네트워크 장애 시 보여 줄 캡처나 녹화본을 준비합니다.
- 모델이 틀리는 대표 사례와 대응을 숨기지 않습니다.
- 비용, 지연, 평가 데이터 수를 숫자로 제시합니다.

## 10분 발표의 논리

```text
문제와 사용자 → 기존 방식의 한계 → 데이터와 기준선
→ 시스템 구조 → 평가 결과 → 실패 사례와 안전
→ 실제 시연 → 다음 개선 계획
```

기술 목록보다 문제, 근거, 결과와 한계를 연결하는 것이 중요합니다.
