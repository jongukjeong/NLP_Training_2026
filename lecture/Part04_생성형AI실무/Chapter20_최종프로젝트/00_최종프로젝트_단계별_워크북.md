## 난이도와 Solution 공개 원칙

- 기본 요구사항을 먼저 작은 단계로 나누어 실행합니다.
- 함수화, 타입 힌트, 복합 평가와 운영 기능은 선택 확장입니다.
- solution은 직접 시도하고 공통 오류 피드백을 받은 뒤 공개합니다.
- 처음부터 solution과 같은 구조로 작성하는 것은 목표가 아닙니다.
<!-- END: BEGINNER_LEARNING_PATH -->

## 1단계: 문제 한 문장

다음 빈칸을 채웁니다.

```text
[사용자]가 [상황]에서 [현재 문제]를 겪을 때,
[입력]을 받아 [검증 가능한 출력]을 제공하여
[측정 지표]를 [목표값]까지 개선한다.
```

예: 상담원이 통화 중 정책을 찾는 시간을 줄이기 위해 질문을 받아 출처 포함 답을 제공하고 Hit@3 0.85와 P95 5초를 달성한다.

## 2단계: 하지 않을 일

범위를 명확히 하기 위해 제외 항목도 씁니다.

- 자동 환불 실행은 하지 않음
- 법률 판단은 하지 않음
- 근거 없는 일반 대화는 지원하지 않음
- 관리자 전용 문서는 검색하지 않음

## 3단계: 데이터 명세

| 항목 | 기록 |
|---|---|
| 출처와 소유자 | 작성 |
| 라이선스·사용권 | 작성 |
| 문서 수와 기간 | 작성 |
| 개인정보 | 작성 |
| 업데이트 주기 | 작성 |
| train/valid/test 분할 | 작성 |

같은 문서의 복제본이나 청크가 서로 다른 분할에 들어가지 않게 합니다.

## 4단계: 기준선

RAG라면 키워드 검색, 분류라면 다수 클래스 또는 TF-IDF 선형 모델을 먼저 만듭니다. 복잡한 모델은 기준선보다 어떤 면이 좋아졌는지 보여야 합니다.

## 5단계: 아키텍처

```text
사용자 → UI/API → 입력 검증 → 검색/모델
→ 출력 검증 → 응답
              ↘ 로그·평가·모니터링
```

각 상자마다 입력, 출력, 실패 처리와 담당자를 적습니다.

## 6단계: 평가셋

전형, 경계, 오타, 긴 입력, 근거 없음, 권한 없음, 공격 입력을 포함합니다. 개발자가 만든 쉬운 질문만 사용하지 않습니다.

## 7단계: 지표

| 영역 | 지표 예 |
|---|---|
| 검색 | Hit@k, MRR |
| 분류 | Macro F1, 클래스 Recall |
| 생성 | 사실성, 근거성, 형식 준수 |
| 운영 | P50/P95 지연, 오류율 |
| 비용 | 요청당 비용, 월 예상 비용 |
| 안전 | 개인정보·권한 위반 건수 |

## 8단계: 비용 계산

$$
월비용=월요청수\times 요청당평균비용+고정인프라비용
$$

하루 3,000건, 건당 20원이라면 30일 사용 비용은 약 180만원이며 서버 고정비는 별도입니다.

## 9단계: 실험 기록

| 실험 ID | 변경 한 가지 | 품질 | P95 | 비용 | 결론 |
|---|---|---:|---:|---:|---|
| base | 기준선 | 기록 | 기록 | 기록 | 기준 |
| exp01 | chunk 300→600 | 기록 | 기록 | 기록 | 채택/기각 |

한 번에 여러 설정을 바꾸지 않습니다.

## 10단계: 실패 사례

최소 20개의 실패를 검색 실패, 생성 오류, 데이터 오류, 권한 오류, 지연·비용 문제로 분류합니다. 성공 화면만 발표하지 않습니다.

## 11단계: 서비스 준비

- 환경변수와 비밀키 분리
- 입력 크기와 rate limit
- timeout과 제한 재시도
- 개인정보 로그 마스킹
- 모델·프롬프트·데이터 버전
- rollback 절차
- 사용자 피드백 수집

## 12단계: 발표 순서

문제와 사용자 → 데이터 → 기준선 → 설계 → 실험 → 정량 결과 → 실제 실패 → 비용·안전 → 다음 개선 순으로 발표합니다.
